`vi` (Visual Editor) is a powerful text editor available in almost all Linux/Unix systems.
`vim` (Vi Improved) is the enhanced version of `vi`.

It is widely used in:
* Embedded Linux development
* Shell scripting
* Kernel development
* Configuration file editing
* Remote server editing through SSH
## Opening a File in Vi
```

```

```

```

* Creates new file if not present

# GCC compiler
The GNU Compiler Collection (GCC) is a compiler system used to compile programs written in languages like:
* C
* C++
* Java
* Fortran
* Go

In development and embedded Linux systems, GCC is mainly used for compiling C/C++ programs.
## What is a Compiler?
A compiler converts:
```
High-Level Language  →  Machine Code
```
Example:
```c
printf("Hello");
```
is converted into executable binary instructions.
## Basic GCC Compilation Process
The GCC compilation process has 4 stages:
```
Source Code
   ↓
Preprocessor
   ↓
Compiler
   ↓
Assembler
   ↓
Linker
   ↓
Executable File
```
## GCC Compilation Stages
### 1. Preprocessing
Processes:
* `#include`
* `#define`
* Macros
* Header files

Command:
```
gcc -E test.c -o test.i
```
Output:
```
test.i
```
(preprocessed file)
## 2. Compilation
Converts C code into assembly code.

Command:
```
gcc -S test.i -o test.s
```
Output:
```
test.s
```
(assembly file)
## 3. Assembly
Converts assembly code into object code.

Command:
```
gcc -c test.s -o test.o
```
Output:
```
test.o
```
(object file)
## 4. Linking
Combines object files and libraries.

Command:
```
gcc test.o -o test
```
Output:
```
Executable file
```

# Four Stages of GCC
The GCC compiler converts a C program into an executable file in **4 stages**.
## GCC Compilation Flow
```
Source Code (.c)
       ↓
1. Preprocessor
       ↓
2. Compiler
       ↓
3. Assembler
       ↓
4. Linker
       ↓
Executable File
```
## Example Program
### test.c
```c
#include<stdio.h>
#define VALUE 10
int main()
{
    printf("%d", VALUE);
    return 0;
}
```
## 1. Preprocessing Stage
The preprocessor handles:
* Header file inclusion (`#include`)
* Macro expansion (`#define`)
* Conditional compilation

Input:
```
test.c
```
Output:
```
test.i
```
(preprocessed file)
## Command
```
gcc -E test.c -o test.i
```
* `-E` → preprocess only
## What Happens Internally
### Before Preprocessing
```c
#define VALUE 10
printf("%d", VALUE);
```
### After Preprocessing
```c
printf("%d", 10);
```
Also:
```c
#include<stdio.h>
```
is expanded into the complete header content.
## Preprocessor Flow
```
Source File (.c)
       ↓
Header Inclusion
Macro Expansion
Conditional Compilation
       ↓
Preprocessed File (.i)
```
## 2. Compilation Stage
The compiler converts:
```
Preprocessed code → Assembly code
```
Input:
```
test.i
```
Output:
```
test.s
```
(assembly file)
## Command
```
gcc -S test.i -o test.s
```
* `-S` → generate assembly code
## Example Assembly Code
```assembly
movl $10, %eax
```
Assembly language is CPU architecture dependent.
## Compilation Flow
```
Preprocessed File (.i)
       ↓
Syntax Checking
Semantic Analysis
Optimization
Code Generation
       ↓
Assembly File (.s)
```
## 3. Assembly Stage
The assembler converts:
```
Assembly code → Machine/Object code
```
Input:
```
test.s
```
Output:
```
test.o
```
(object file)
## Command
```
gcc -c test.s -o test.o
```
* `-c` → compile only (no linking)
## What is Object File?
Object file contains:
* Machine code
* Symbol table
* Relocation information

But it is NOT executable yet.
## Assembly Flow
```
Assembly File (.s)
       ↓
Assembler
       ↓
Object File (.o)
```
## 4. Linking Stage
The linker combines:
* Object files
* Library files

to create executable file.

Input:
```
test.o
```
Output:
```
a.out or executable file
```
## Command
```
gcc test.o -o test
```
OR directly:
```
gcc test.c -o test
```
## What Happens During Linking?
The linker resolves external functions like:
```c
printf()
```
from standard C libraries.

Without linking:
```
Undefined reference to printf
```
error occurs.
## Linking Flow
```
Object File (.o)
       ↓
Link Standard Libraries
Resolve Symbols
Combine Multiple Objects
       ↓
Executable File
```
## Complete GCC Stage Commands
| Stage         | Command                   | Output     |
| ------------- | ------------------------- | ---------- |
| Preprocessing | `gcc -E test.c -o test.i` | `.i`       |
| Compilation   | `gcc -S test.i -o test.s` | `.s`       |
| Assembly      | `gcc -c test.s -o test.o` | `.o`       |
| Linking       | `gcc test.o -o test`      | Executable |
## File Extensions in GCC Stages
| Extension  | Meaning           |
| ---------- | ----------------- |
| `.c`       | Source code       |
| `.i`       | Preprocessed code |
| `.s`       | Assembly code     |
| `.o`       | Object code       |
| Executable | Final program     |
## Internal GCC Pipeline
```
test.c
  ↓
cpp      → Preprocessor
  ↓
cc1      → Compiler
  ↓
as       → Assembler
  ↓
ld       → Linker
  ↓
Executable
```

| Compiler                           | Linker                    |
| ---------------------------------- | ------------------------- |

# Functionality of Preprocessor
The **Preprocessor** is the first stage of the GCC compilation process.

It processes all **preprocessor directives** before actual compilation starts.

Preprocessor directives begin with:
```c 
#
```
Example:
```c 
#include<stdio.h>
#define SIZE 10
```
## Role of Preprocessor
The preprocessor performs:
1. Header file inclusion
2. Macro expansion
3. Conditional compilation
4. File inclusion
5. Token manipulation
## Preprocessor Flow
```
Source File (.c)
       ↓
Preprocessor
       ↓
Expanded Source File (.i)
```
## Preprocessor Command
```
gcc -E test.c -o test.i
```
* `-E` → preprocess only
* Output file → `.i`
## Main Functionalities of Preprocessor
| Functionality           | Purpose                 |
| ----------------------- | ----------------------- |
| `#include`              | Include header files    |
| `#define`               | Define macros           |
| `#undef`                | Remove macro definition |
| Conditional Compilation | Compile selected code   |
| Macro Substitution      | Replace symbols         |
| Token Pasting           | Combine tokens          |
| Stringification         | Convert token to string |
## 1. Header File Inclusion
Used to include library or user-defined headers.
### Syntax
```c 
#include<stdio.h>
#include "myheader.h"
```
## Types of Header Files
| Type          | Example     | Search Location    |
| ------------- | ----------- | ------------------ |
| System Header | `<stdio.h>` | System directories |
| User Header   | `"my.h"`    | Current directory  |
## Example
### Before Preprocessing
```c 
#include<stdio.h>
int main()
{
    printf("Hello");
}
```
### After Preprocessing
The content of `stdio.h` is inserted into source code.
## 2. Macro Definition (`#define`)

# Functionality of Compiler
The **Compiler** is the second stage of the GCC compilation process.

It converts:
```
Preprocessed Source Code (.i)
            ↓
Assembly Code (.s)
```
The compiler checks the correctness of the program and generates optimized assembly code.
## Position of Compiler in GCC Stages
```
Source Code (.c)
      ↓
Preprocessor
      ↓
Preprocessed File (.i)
      ↓
Compiler
      ↓
Assembly File (.s)
      ↓
Assembler
      ↓
Object File (.o)
      ↓
Linker
      ↓
Executable
```
## Compiler Command
```
gcc -S test.i -o test.s
```
OR directly:
```
gcc -S test.c
```
* `-S` → generates assembly code only
## Main Functionalities of Compiler
| Functionality                | Purpose                         |
| ---------------------------- | ------------------------------- |
| Lexical Analysis             | Converts characters into tokens |
| Syntax Analysis              | Checks grammar/syntax           |
| Semantic Analysis            | Checks meaning/type correctness |
| Intermediate Code Generation | Internal representation         |
| Optimization                 | Improves performance            |
| Code Generation              | Produces assembly code          |
| Error Detection              | Finds compilation errors        |
## Compiler Internal Phases
```
Source Code
    ↓
Lexical Analysis
    ↓
Syntax Analysis
    ↓
Semantic Analysis
    ↓
Intermediate Code Generation
    ↓
Code Optimization
    ↓
Assembly Code Generation
```
## 1. Lexical Analysis
Also called:
```
Scanner / Tokenizer
```
It converts source code into tokens.
## What is a Token?
Smallest meaningful unit in a program.

Example:
```c 
int a = 10;
```
Tokens:
| Token | Type           |
| ----- | -------------- |
| `int` | Keyword        |
| `a`   | Identifier     |
| `=`   | Operator       |
| `10`  | Constant       |
| `;`   | Special symbol |
## Lexical Analysis Flow
```
Characters
    ↓
Tokenizer
    ↓
Tokens
```
## Errors Detected
| Error Type      | Example   |
| --------------- | --------- |
| Invalid symbols | `int @a;` |
| Invalid tokens  | `12abc`   |
## 2. Syntax Analysis
Also called:
```
Parser
```
Checks grammatical correctness using C language rules.
## Example
Correct:
```c 
int a;
```
Incorrect:
```c 
int ;
```
Compiler reports:
```
Syntax Error
```
## Syntax Tree
The parser creates:
```
Parse Tree / Syntax Tree
```
for program structure.
## Syntax Analysis Flow
```
Tokens
   ↓
Parser
   ↓
Syntax Tree
```
## 3. Semantic Analysis
Checks meaning and logical correctness.
## Semantic Checks
| Check                | Example             |
| -------------------- | ------------------- |
| Type checking        | `int + char`        |
| Variable declaration | undeclared variable |
| Function arguments   | wrong parameters    |
| Scope checking       | invalid access      |
## Example
```c
int a;
a = "Hello";
```
Compiler error:
```
Type mismatch
```
## Another Example
```c 
printf("%d", x);
```
If `x` not declared:
```
Undeclared variable error
```
## 4. Intermediate Code Generation
Compiler generates intermediate representation (IR).

Example:
```c 
a = b + c;
```
Intermediate code:
```
ADD b, c
STORE a
```
This makes optimization easier.
## 5. Code Optimization
Improves:
* Speed
* Memory usage
* Performance

without changing program output.
## Types of Optimizations
| Optimization          | Purpose                   |
| --------------------- | ------------------------- |
| Constant folding      | Calculate constants early |
| Dead code elimination | Remove unused code        |
| Loop optimization     | Faster loops              |
| Inline expansion      | Replace function calls    |
## Example: Constant Folding
Before optimization:
```c 
int x = 10 * 20;
```
After optimization:
```c 
int x = 200;
```
## Example: Dead Code Elimination
```c
return 0;
printf("Hello");
```
`printf()` is removed because it never executes.
## GCC Optimization Levels
| Option | Meaning               |
| ------ | --------------------- |
| `-O0`  | No optimization       |
| `-O1`  | Basic optimization    |
| `-O2`  | Moderate optimization |
| `-O3`  | High optimization     |
Example:
```
gcc -O2 test.c
```
## 6. Code Generation
Compiler converts optimized code into:
```
Assembly Language
```
Output file:
```
.s
```
## Example Assembly Output
For:
```c 
int a = 10;
```
Compiler may generate:
```assembly

```
## Compiler Error Detection
The compiler detects:
| Error Type           | Example            |
| -------------------- | ------------------ |
| Syntax errors        | Missing semicolon  |
| Type errors          | Wrong data type    |
| Undeclared variables | Unknown identifier |
| Function mismatch    | Wrong arguments    |
## Example Compilation Error
```c

{
    int a
    return 0;
}
```
Error:
```

```
## Compiler vs Interpreter
| Compiler                       | Interpreter              |
| ------------------------------ | ------------------------ |
| Translates whole program       | Executes line-by-line    |
| Faster execution               | Slower                   |
| Generates executable           | No executable            |
| Errors shown after compilation | Errors shown immediately |
## Real-Time Compiler Usage
### Embedded Linux
```

```

1. Detects errors before execution
2. Generates efficient machine code
3. Improves performance using optimization
4. Produces executable programs
5. Supports debugging information
## Disadvantages

# Functionality of Assembler
The **Assembler** is the third stage of the GCC compilation process.

It converts:
```
Assembly Code (.s)
        ↓
Object Code (.o)
```
The assembler translates assembly language instructions into machine-level binary code understood by the processor.
## Position of Assembler in GCC Stages
```
Source Code (.c)
      ↓
Preprocessor
      ↓
Preprocessed File (.i)
      ↓
Compiler
      ↓
Assembly File (.s)
      ↓
Assembler
      ↓
Object File (.o)
      ↓
Linker
      ↓
Executable File
```
## Assembler Command
```
gcc -c test.s -o test.o
```
OR
```
as test.s -o test.o
```
* `-c` → compile only (no linking)
## Input and Output of Assembler
| Input              | Output           |
| ------------------ | ---------------- |
| `.s` Assembly file | `.o` Object file |
## Main Functionalities of Assembler
| Functionality                   | Purpose                     |
| ------------------------------- | --------------------------- |
| Translate Assembly Instructions | Convert to machine code     |
| Generate Object File            | Create `.o` file            |
| Assign Memory Addresses         | Address allocation          |
| Create Symbol Table             | Store symbols/functions     |
| Generate Relocation Info        | For linker usage            |
| Detect Assembly Errors          | Invalid instructions/syntax |
## Assembly Language Example
### Input File: test.s
```assembly
.section .text
.globl main
main:
    movl $10, %eax
    ret
```
## What Assembler Does
The assembler converts:
```assembly 
movl $10, %eax
```  
into machine code:
```
B8 0A 00 00 00
```
(Binary/hexadecimal representation)
## Assembly Process Flow
```
Assembly Code (.s)
        ↓
Assembler
        ↓
Machine Instructions
        ↓
Object File (.o)
```
## What is an Object File?
An object file contains:
* Machine code
* Symbol table
* Relocation information
* Debugging information (optional)

But it is NOT executable yet.
## Structure of Object File
```
Object File (.o)
 ├── Header
 ├── Text Section
 ├── Data Section
 ├── Symbol Table
 ├── Relocation Table
 └── Debug Information
```
## Important Sections in Object File
| Section   | Purpose                      |
| --------- | ---------------------------- |
| `.text`   | Executable instructions      |
| `.data`   | Initialized global variables |
| `.bss`    | Uninitialized variables      |
| `.rodata` | Read-only data               |
## Example Program
### test.c
```c 
int main()
{
    return 10;
}
```
## Generate Assembly File
```

```
Produces:
```

```

Produces:
```
test.o
```

### Using `objdump`

Displays disassembled machine code.

```

```

## Example Output of `nm`
```

```
* `T` → symbol in text section
## Symbol Table
The assembler creates a symbol table containing:
* Function names
* Variable names
* Labels

Used later by linker.
## Example
```assembly
main:
```
`main` becomes a symbol entry.
## Relocation Information
Some addresses are unknown during assembly.

Example:
```c 
printf("Hello");
```
The actual address of `printf()` is resolved later by the linker.

Assembler stores relocation entries for such symbols.
## Relocation Flow
```
Assembler
    ↓
Unknown External Symbols
    ↓

    ↓
Handled by Linker
```
## Errors Detected by Assembler
| Error Type          | Example                   |
| ------------------- | ------------------------- |


```

```

| Compiler               | Assembler                        |
| ---------------------- | -------------------------------- |
| Converts C → Assembly  | Converts Assembly → Machine code |
| Generates `.s`         | Generates `.o`                   |
| High-level translation | Low-level translation            |
## Assembler vs Linker
| Assembler                   | Linker                    |
| --------------------------- | ------------------------- |
| Creates object files        | Creates executable        |

4. Enables hardware interaction
5. Prepares code for linking

2. Difficult to write manually
3. Low-level programming complexity

# Functionality of Linker
The **Linker** is the fourth and final stage of the GCC compilation process.

It combines:
* Object files (`.o`)
* Library files (`.a`, `.so`)

and produces the final:
```
Executable File
```
## Position of Linker in GCC Stages
```
Source Code (.c)
      ↓
Preprocessor
      ↓
Compiler
      ↓
Assembler
      ↓
Object File (.o)
      ↓
Linker
      ↓
Executable File
```
## Role of Linker
The linker performs:
1. Combining object files
2. Resolving external symbols
3. Linking libraries
4. Address allocation
5. Creating executable file
## Linking Command
```
gcc main.o display.o -o app
```
OR directly:
```
gcc main.c display.c -o app
```
## Input and Output of Linker
| Input             | Output          |
| ----------------- | --------------- |
| `.o`, `.a`, `.so` | Executable file |
## Main Functionalities of Linker
| Functionality           | Purpose                            |
| ----------------------- | ---------------------------------- |
| Symbol Resolution       | Match function/variable references |
| Address Binding         | Assign memory addresses            |
| Library Linking         | Connect external libraries         |
| Object File Combination | Merge multiple object files        |
| Executable Creation     | Produce runnable program           |
| Relocation              | Adjust addresses                   |
## Example of Linking
### main.c
```c

```


```

```
Creates executable:
```
app
```
## What Happens During Linking?
The linker:
1. Combines machine code from `.o` files
2. Resolves function calls
3. Resolves library references
4. Creates executable layout
## Symbol Resolution
Suppose:
```c

```
exists in `main.o`.

The linker searches:
```

```
for the actual definition.
## Symbol Resolution Flow
```

```

# Interrupting the Compiler
“Interrupting the compiler” means stopping the GCC compilation process at a specific stage instead of completing all four stages.

Normally GCC performs:
```
Preprocessing
    ↓
Compilation
    ↓
Assembly
    ↓
Linking
```
But GCC provides options to stop after any stage.
## Why Interrupt the Compiler?
Interrupting compilation is useful for:
* Debugging compilation issues
* Viewing intermediate files
* Analyzing assembly code
* Separate compilation
* Optimizing builds
* Embedded development
## GCC Compilation Stages
```
test.c
   ↓
Preprocessor
   ↓
test.i
   ↓
Compiler
   ↓
test.s
   ↓
Assembler
   ↓
test.o
   ↓
Linker
   ↓
Executable
```
## GCC Options to Interrupt Compilation
| Option | Stops After   | Output |
| ------ | ------------- | ------ |
| `-E`   | Preprocessing | `.i`   |
| `-S`   | Compilation   | `.s`   |
| `-c`   | Assembly      | `.o`   |
## 1. Interrupt After Preprocessing (`-E`)
Stops after preprocessor stage.
## Command
```
gcc -E test.c -o test.i
```
## Output
```
test.i
```
(preprocessed file)
## What Happens?
The preprocessor expands:
* `#include`
* `#define`
* macros
* conditional compilation
## Example
### Source Code
```c 
#include<stdio.h>
#define MAX 100
int main()
{
    printf("%d", MAX);
}
```
## After Preprocessing
```c 
printf("%d", 100);
```
## Use Cases of `-E`
| Use                           |
| ----------------------------- |
| Debug macros                  |
| Verify header inclusion       |
| Check conditional compilation |
| Analyze expanded source       |
## 2. Interrupt After Compilation (`-S`)
Stops after compiler stage.
## Command
```
gcc -S test.c -o test.s
```
## Output
```
test.s
```
(assembly file)
## What Happens?
Compiler converts:
```
C Code → Assembly Code
```
## Example Assembly
```
movl $10, %eax
```
## Use Cases of `-S`
| Use                          |
| ---------------------------- |
| Study assembly language      |
| Analyze optimization         |
| Debug low-level code         |
| Embedded systems development |
## 3. Interrupt After Assembly (`-c`)
Stops after assembler stage.
## Command
```
gcc -c test.c -o test.o
```
## Output
```
test.o
```
(object file)
## What Happens?
Assembler converts:
```
Assembly Code → Machine/Object Code
```
## Use Cases of `-c`
| Use                       |
| ------------------------- |
| Separate compilation      |
| Faster incremental builds |
| Library creation          |
| Modular development       |

### main.c

```c 
void display();

    display();
}
```

```c 
{
```

```
gcc -c main.c

```
Produces:
```
main.o
display.o
```
## Link Later
```

```

```
gcc -E → Stops after preprocessing

```

```
   ↓
```

# Compiling a C Program
Compiling a C program means converting C source code into an executable program using a compiler such as GCC.
## Steps in Compiling a C Program
The GCC compiler performs 4 stages
```
C Source Code (.c)
        ↓
1. Preprocessor
        ↓
2. Compiler
        ↓
3. Assembler
        ↓
4. Linker
        ↓
Executable File
```
## Example C Program
### hello.c
```c 
#include<stdio.h>
int main()
{
    printf("Hello World\n");
    return 0;
}
```
## Basic Compilation Command
```
gcc hello.c -o hello
```
## Explanation
| Part      | Meaning         |
| --------- | --------------- |
| `gcc`     | GNU C Compiler  |
| `hello.c` | Source file     |
| `-o`      | Output option   |
| `hello`   | Executable name |
## Run the Program
```
./hello
```
## Output
```
Hello World
```
## Compilation Without `-o`
```
gcc hello.c
```
Default executable created:
```
a.out
```
Run:
```
./a.out
```
## Internal GCC Compilation Stages
## 1. Preprocessing Stage
Processes:
* `#include`
* `#define`
* macros

Command:
```
gcc -E hello.c -o hello.i
```
Generated file:
```
hello.i
```
(preprocessed file)
## 2. Compilation Stage
Converts C code into assembly code.
Command:
```
gcc -S hello.i -o hello.s
```
Generated file:
```
hello.s
```
(assembly file)
## 3. Assembly Stage
Converts assembly into object code.

Command:
```
gcc -c hello.s -o hello.o
```
Generated file:
```
hello.o
```
(object file)
## 4. Linking Stage
Combines object files and libraries.

Command:
```
gcc hello.o -o hello
```
Generated file:
```
Executable file
```
## Complete Compilation Flow
```
hello.c
   ↓
gcc -E
   ↓
hello.i
   ↓
gcc -S
   ↓
hello.s
   ↓
gcc -c
   ↓
hello.o
   ↓
gcc
   ↓
hello (Executable)
```

# Preprocessor Features
The **C Preprocessor** is a program that processes source code before actual compilation begins.

It works on all lines starting with:
```c 
#
```
called **preprocessor directives**.
## Position of Preprocessor in GCC
```
Source File (.c)
      ↓
Preprocessor
      ↓
Expanded File (.i)
      ↓
Compiler
```
## Main Features of Preprocessor
| Feature                 | Purpose                      |
| ----------------------- | ---------------------------- |
| Macro Expansion         | Replace symbols with values  |
| File Inclusion          | Include header files         |
| Conditional Compilation | Compile selected code        |
| Token Pasting           | Merge tokens                 |
| Stringification         | Convert token to string      |
| Header Guards           | Prevent multiple inclusion   |
| Built-in Macros         | Provide compiler information |
## 1. Macro Expansion
Macros are defined using:
```c 
#define
```
## Object-like Macro
### Example
```c 
#define MAX 100
```
Usage:
```c 
printf("%d", MAX);
```
After preprocessing:
```c
printf("%d", 100);
```
## Function-like Macro
### Example
```c 
#define SQUARE(x) ((x)*(x))
```
Usage:
```c 
SQUARE(5)
```
Expands to:
```c
((5)*(5))
```
## Advantages of Macros
* Faster than function calls
* Code reuse
* Easy constants definition
## 2. File Inclusion
Used to include contents of another file.
## Syntax
### System Header
```c
#include<stdio.h>
```
### User Header
```c 
#include "myheader.h"
```
## Purpose
* Reuse declarations
* Access library functions
* Modular programming
## Example
```c 
#include<stdio.h>
int main()
{
    printf("Hello");
}
```
The preprocessor inserts the contents of:
```
stdio.h
```
into the program.
## 3. Conditional Compilation
Compiles code only if specified condition is true.
## Directives Used
| Directive | Purpose              |
| --------- | -------------------- |
| `#if`     | Checks condition     |
| `#ifdef`  | If macro defined     |
| `#ifndef` | If macro not defined |
| `#else`   | Alternative code     |
| `#elif`   | Else-if              |
| `#endif`  | End condition        |
## Example: `#ifdef`
```c
#define DEBUG
#ifdef DEBUG
printf("Debug Mode");
#endif
```
If `DEBUG` defined:
```
Debug Mode
```
gets compiled.
## Example: `#ifndef`
```c 
#ifndef HEADER_H
#define HEADER_H
#endif
```
Used as:
```
Header Guard
```
## 4. Header Guards
Prevent multiple inclusion of same header file.
## Problem Without Header Guards
If same header included multiple times:
```c 
#include "a.h"
#include "a.h"
```
duplicate declarations may occur.
## Solution
```c 
#ifndef FILE_H
#define FILE_H
// declarations
#endif
```
## Header Guard Flow
```
Check if macro defined
        ↓
Not Defined
        ↓
Define Macro
        ↓
Include Header Content
```
## 5. Token Pasting Operator (`##`)
Combines two tokens into one.
## Example
```c 
#define MERGE(a,b) a##b
```
Usage:
```c 
MERGE(num,1)
```
Expands to:
```c
num1
```
## Uses
* Variable generation
* Generic macros
* Embedded register names
## 6. Stringification Operator (`#`)
Converts macro argument into string.
## Example
```c
#define PRINT(x) #x
```
Usage:
```c
PRINT(Hello)
```
Expands to:
```c 
"Hello"
```
## Uses
* Debugging macros
* Logging systems
* Diagnostic messages
## 7. Built-in Macros
Predefined by compiler.
## Important Built-in Macros
| Macro      | Meaning               |
| ---------- | --------------------- |
| `__FILE__` | Current filename      |
| `__LINE__` | Current line number   |
| `__DATE__` | Compilation date      |
| `__TIME__` | Compilation time      |
| `__func__` | Current function name |
## Example
```c 
printf("%s", __FILE__);
```
Output:
```

```
## 8. `#undef`
Removes a macro definition.
## Example
```c 
#define SIZE 100
#undef SIZE
```
Now `SIZE` is undefined.
## 9. Nested Macros
Macros can contain other macros.
## Example
```c 
#define A 10

```
`B` expands to:
```c 
10+5
```
## 10. Multiline Macros
Use `\` for continuation.
## Example
```c

printf("Hello\n"); \
printf("World\n");
```
## Preprocessor Output File
Generate preprocessed output:
```
gcc -E test.c -o test.i
```
Output file:
```
test.i
```

| ----------------------- | ------------------------- |
| Macros                  | Faster execution          |

# Predefined Preprocessor Symbols
Predefined preprocessor symbols are built-in macros automatically provided by the C compiler.

These macros are expanded by the preprocessor before compilation.

They are mainly used for:
* Debugging
* Logging
* Error reporting
* Conditional compilation
* Compiler information
## What are Predefined Macros?
They are special macros already defined by the compiler.

Example:
```c 
__LINE__
```
The compiler replaces it with the current line number.
## Important Predefined Preprocessor Symbols
| Macro              | Meaning                 |
| ------------------ | ----------------------- |
| `__FILE__`         | Current source filename |
| `__LINE__`         | Current line number     |
| `__DATE__`         | Compilation date        |
| `__TIME__`         | Compilation time        |
| `__STDC__`         | ANSI C compliance       |
| `__func__`         | Current function name   |
| `__STDC_VERSION__` | C standard version      |
## 1. `__FILE__`
Represents:
```
Current source file name
```
## Example
```c 
#include<stdio.h>
int main()
{
    printf("%s", __FILE__);
}
```
## Output
```
test.c
```
## Uses
* Error reporting
* Debug logging
* Trace information
## 2. `__LINE__`
Represents:
```
Current line number
```
## Example
```c 
#include<stdio.h>
int main()
{
    printf("%d", __LINE__);
}
```
## Output Example
```
5
```
(line number depends on code position)
## Uses
* Debugging
* Error tracking
* Log messages
## 3. `__DATE__`
Represents:
```
Compilation date
```
Format:
```
"Mmm dd yyyy"
```
## Example
```c 
#include<stdio.h>
int main()
{
    printf("%s", __DATE__);
}
```
## Output Example
```
May 18 2026
```
## Uses
* Build information
* Software version tracking
## 4. `__TIME__`
Represents:
```
Compilation time
```
Format:
```
"hh:mm:ss"
```
## Example
```c 
#include<stdio.h>
int main()
{
    printf("%s", __TIME__);
}
```
## Output Example
```
10:45:20
```
## Uses
* Build timestamp
* Logging
## 5. `__STDC__`
Indicates whether compiler follows ANSI C standard.

```c 
#include<stdio.h>
int main()
{
    printf("%d", __STDC__);
}
```
## Output
```

```
Meaning:
```
ANSI C compliant compiler
```

Represents:
```
Current function name
```

## Output
```
display
```
## Uses
* Function tracing
* Debugging
* Logging

Represents C standard version.

```c
#include<stdio.h>
```

# Warnings and Extensions
GCC provides:
1. **Warnings** → Help identify possible problems in code
2. **Extensions** → Additional features beyond standard C

These improve:
* Code quality
* Debugging
* Performance
* Compiler flexibility
## 1. Warnings in GCC
Warnings are messages generated by the compiler indicating:
```
Possible mistakes in the program
```
The program may still compile successfully.
## Why Warnings are Important?
Warnings help detect:
* Logical mistakes
* Uninitialized variables
* Type mismatches
* Unused variables/functions
* Dangerous code
## Example Program with Warning
```c 
#include<stdio.h>
int main()
{
    int a;
    printf("%d", a);
    return 0;
}
```
## Compilation
```
gcc -Wall test.c
```
## Warning Message
```
warning: ‘a’ is used uninitialized
```
## Common GCC Warning Options
| Option         | Meaning                       |
| -------------- | ----------------------------- |
| `-Wall`        | Enable common warnings        |
| `-Wextra`      | Extra warnings                |
| `-Werror`      | Treat warnings as errors      |
| `-pedantic`    | Strict ANSI standard checking |
| `-Wunused`     | Warn about unused variables   |
| `-Wshadow`     | Warn about variable shadowing |
| `-Wconversion` | Warn about type conversion    |
| `-Wformat`     | Format string checking        |
## 1. `-Wall`
Enables most commonly used warnings.
## Example
```
gcc -Wall test.c
```
## Detects
* Uninitialized variables
* Unused variables
* Missing return statements
* Incorrect types
## 2. `-Wextra`
Provides additional warnings.
## Example
```
gcc -Wall -Wextra test.c
```

## Example
```
gcc -Wall -Werror test.c
```

## Example Output
```
error: variable ‘a’ is uninitialized
```

## 4. `-pedantic`
Checks strict ANSI/ISO C compliance.
## Example
```
gcc -pedantic test.c
```
## Use
* Portable code development
* Standard-compliant programming

# Optimization
Optimization is the process of improving the performance and efficiency of a program without changing its output.

The GCC compiler performs optimization to:
* Increase execution speed
* Reduce memory usage
* Reduce code size
* Improve CPU utilization
## Purpose of Optimization
Optimization helps in:
| Goal                    | Benefit                  |
| ----------------------- | ------------------------ |
| Faster execution        | Better performance       |
| Smaller binaries        | Reduced memory usage     |
| Efficient CPU usage     | Improved responsiveness  |
| Lower power consumption | Embedded systems benefit |
## GCC Optimization Stages
Optimization is mainly performed during:
```
Compiler Stage
```
before generating assembly code.
## Basic Optimization Command
```
gcc -O2 test.c -o test
```
## GCC Optimization Levels
| Option   | Meaning                 |
| -------- | ----------------------- |
| `-O0`    | No optimization         |
| `-O1`    | Basic optimization      |
| `-O2`    | Moderate optimization   |
| `-O3`    | High optimization       |
| `-Os`    | Optimize for size       |
| `-Ofast` | Aggressive optimization |
## 1. `-O0` (No Optimization)
Default mode.
## Features
* Faster compilation
* Easier debugging
* No optimization applied
## Command
```
gcc -O0 test.c
```
## Use Cases
* Debugging
* Development stage
## 2. `-O1` (Basic Optimization)
Applies simple optimizations.
## Features
* Reduces redundant code
* Slight performance improvement
* Moderate compile time
## Command
```
gcc -O1 test.c
```
## 3. `-O2` (Recommended Optimization)
Most commonly used optimization level.
## Features
* Good balance of speed and size
* Safe optimizations
* Better runtime performance
## Command
```
gcc -O2 test.c
```
## Commonly Used In
* Production applications
* Embedded systems
* Linux utilities
## 4. `-O3` (High Optimization)
Performs aggressive optimizations.
## Features
* Loop optimizations
* Function inlining
* Vectorization
## Command
```
gcc -O3 test.c
```
## Advantage
Very fast execution.
## Disadvantage
* Larger executable size
* Difficult debugging
## 5. `-Os` (Optimize for Size)
Optimizes code size rather than speed.
## Command
```
gcc -Os test.c
```
## Uses
* Embedded systems
* Memory-constrained devices
## 6. `-Ofast`
Aggressive optimization ignoring some standards.
## Command
```
gcc -Ofast test.c
```
## Features
* Maximum speed
* May violate strict IEEE standards
## Types of Optimizations
## 1. Constant Folding
Compiler computes constants at compile time.
## Example
Before optimization:
```c 
int x = 10 * 20;
```
After optimization:
```c
int x = 200;
```
## Benefit
Reduces runtime calculations.
## 2. Constant Propagation
Replaces variables with constant values.
## Example
```c 
int a = 10;
int b = a + 5;
```
Optimized to:
```c 
int b = 15;
```
## 3. Dead Code Elimination
Removes unreachable or unused code.
## Example
```c 
return 0;
printf("Hello");
```
`printf()` is removed.
## Benefit
Smaller executable.
## 4. Loop Optimization
Improves loop efficiency.
## Techniques
| Technique             | Purpose                                 |
| --------------------- | --------------------------------------- |
| Loop unrolling        | Reduce loop overhead                    |
| Loop fusion           | Merge loops                             |
| Loop invariant motion | Move constant calculations outside loop |
## Example
Before:
```c 
for(i=0;i<100;i++)
{
    x = a*b;
}
```
After optimization:
```c 
temp = a*b;
for(i=0;i<100;i++)
{
    x = temp;
}
```
## 5. Inline Expansion
Replaces function call with actual function code.
## Example
```c 
inline int add(int a,int b)
{
    return a+b;
}
```
Compiler inserts function body directly.
## Benefit
Removes function call overhead.
## 6. Strength Reduction
Replaces expensive operations with cheaper ones.
## Example
Before:
```c 
x = y * 2;
```
After:
```c 
x = y << 1;
```
## 7. Common Subexpression Elimination
Avoids repeated calculations.
## Example
Before:
```c
a = b*c + d;
x = b*c + e;
```
After:
```c 
temp = b*c;
a = temp + d;
x = temp + e;
```
## Optimization and Debugging
Optimization can make debugging difficult because:
* Variables may disappear
* Instructions reordered
* Functions inlined
## Best Practice
For debugging:
```
gcc -O0 -g test.c
```
For release:
```
gcc -O2 test.c
```
## View Optimized Assembly
Generate assembly:
```
gcc -S -O2 test.c
```
## Optimization in Embedded Systems
Optimization is very important because embedded systems have:
* Limited RAM
* Limited ROM
* Limited CPU power
## Example
```
arm-linux-gcc -Os firmware.c
```
Used to reduce firmware size.
## Optimization vs Code Size
| Optimization | Effect                       |
| ------------ | ---------------------------- |
| `-O3`        | Faster but larger            |
| `-Os`        | Smaller but sometimes slower |
## Advantages of Optimization
1. Faster execution
2. Smaller binaries
3. Better CPU efficiency
4. Reduced power usage
5. Improved system performance
## Disadvantages of Optimization
1. Harder debugging
2. Longer compilation time
3. Larger executable (sometimes)
4. Complex assembly generation
## Real-Time Uses
| Area                       | Usage                |
| -------------------------- | -------------------- |
| Embedded Systems           | Reduce firmware size |
| Linux Kernel               | Improve performance  |
| Drivers                    | Faster execution     |
| High-performance Computing | Maximum speed        |
