# What Is My Program Doing?
In Linux system programming, **“What is my program doing?”** refers to understanding or monitoring what a running program is actually doing internally — such as which functions it is calling, what files it accesses, how it interacts with the operating system, and why it behaves in a certain way.

This is usually done using debugging and tracing tools.
## Why do we need to know what a program is doing?
You may want to inspect a program when:
* The program hangs or becomes slow
* A file is not opening correctly
* A system call fails
* Memory usage becomes too high
* The program crashes unexpectedly
* You want to understand another program’s behavior
### Common ways to see what a program is doing
#### 1. System Call Tracing using `strace`
`strace` shows **system calls** made by a program (interaction between program and kernel).

Example:
```
strace ls
```
Output (simplified):
```
openat(AT_FDCWD, "/etc/ld.so.cache", O_RDONLY) = 3
read(3, "...", 832) = 832
write(1, "file1 file2\n", 12) = 12
```
This tells us:
* `openat()` → opened a file
* `read()` → read data
* `write()` → printed output

To trace an already running process:
```
strace -p 1234
```
(where `1234` is the process ID)
#### 2. Library Call Tracing using `ltrace`
`ltrace` shows **library function calls** (functions from libraries like glibc).

Example:
```
ltrace ./myprogram
```
You may see:
```
printf("Hello World\n")
malloc(128)
free(0x123456)
```
This helps track:
* `printf()`
* `malloc()`
* `free()`
* String operations
#### 3. Debugging with GDB
The GNU Debugger allows you to inspect execution step-by-step.

GDB can help you:
* Pause execution
* View variables
* Set breakpoints
* Execute line by line

Example:
```
gdb ./program
```
Useful commands:
```gdb
break main
run
next
print variable
continue
```
#### 4. Process Monitoring Commands
Linux provides commands to inspect running processes:
* `ps` → show running processes
```
ps aux
```
* `top` → live CPU and memory monitoring
```
top
```
* `htop` → interactive process viewer
```
htop
```
#### 5. Logging and Print Statements
Sometimes the simplest way:
```c
printf("Reached function A\n");
```
or
```c
fprintf(stderr, "Error happened\n");
```
This helps trace execution flow.
### Example Scenario
Suppose this program hangs:
```c
#include <stdio.h>
#include <unistd.h>
int main() {
    printf("Start\n");
    sleep(20);
    printf("End\n");
    return 0;
}
```
Run:
```
strace ./a.out
```
You will notice:
```
write(1, "Start\n", 6) = 6
nanosleep(...)
write(1, "End\n", 4) = 4
```
This tells you the program is sleeping and not frozen.

# Source Level Debugging
**Source-level debugging** means debugging a program by viewing and controlling execution at the **source code level (C/C++ code)** rather than machine instructions or assembly code.

It allows a programmer to execute the program step by step, inspect variables, pause execution, and locate bugs directly in the source code.

The most common debugger used in Linux is GDB.
## Why Source-Level Debugging is Needed
It helps to:
* Find logical and runtime errors
* Track variable values during execution
* Understand program flow
* Detect crashes and segmentation faults
* Debug loops, functions, and conditions
## Features of Source-Level Debugging
### 1. Breakpoints
A **breakpoint** pauses program execution at a specific line or function.

Example:
```c
#include <stdio.h>
int main() {
    int a = 10, b = 20;
    int sum = a + b;
    printf("%d\n", sum);
    return 0;
}
```
Compile with debugging information:
```
gcc -g test.c -o test
```
Start debugger:
```
gdb ./test
```
Set breakpoint:
```gdb
break main
run
```
The program stops at `main()`.
## 2. Step-by-Step Execution
You can execute code line by line.

Common commands:
| Command          | Meaning                 |
| ---------------- | ----------------------- |
| `next` (`n`)     | Execute next line       |
| `step` (`s`)     | Enter function call     |
| `continue` (`c`) | Resume execution        |
| `finish`         | Run until function ends |
Example:
```gdb
next
step
continue
```
## 3. Variable Inspection
You can inspect variable values during execution.

Example:
```gdb
print a
print sum
```
Output:
```
$1 = 10
$2 = 30
```
## 4. Watching Variables
A **watchpoint** stops execution when a variable changes.

Example:
```gdb
watch sum
```
The debugger pauses whenever `sum` changes.
## 5. Stack Inspection
Shows the function call chain.

Example:
```gdb
backtrace
```
or
```gdb
bt
```
Useful for crashes and segmentation faults.
## Steps in Source-Level Debugging
### Step 1: Write Program
```c
#include <stdio.h>
int main() {
    int x = 10;
    int y = 0;
    int z = x / y;
    printf("%d", z);
    return 0;
}
```
This program causes division by zero.
### Step 2: Compile with Debug Symbols
```
gcc -g demo.c -o demo
```
The `-g` option adds debugging information.
### Step 3: Start Debugger
```
gdb ./demo
```
### Step 4: Run Program
```gdb
run
```
Debugger stops at crash location.
### Step 5: Inspect Problem
```gdb
list
print x
print y
bt
```
You can identify that `y = 0` caused the error.
## Important GDB Commands
| Command            | Purpose           |
| ------------------ | ----------------- |
| `run`              | Start program     |
| `break`            | Set breakpoint    |
| `next`             | Execute next line |
| `step`             | Enter function    |
| `continue`         | Resume execution  |
| `print`            | Display variable  |
| `watch`            | Monitor variable  |
| `list`             | Show source code  |
| `backtrace` / `bt` | Show call stack   |
| `quit`             | Exit debugger     |
## Advantages of Source-Level Debugging
* Easy bug detection
* Understands source code directly
* Helps inspect variables and memory
* Faster problem diagnosis
* Useful for large system programs

# Invoking gdb
**Invoking GDB** means starting the debugger to debug a program.

Before using GDB, the program should be compiled with debugging information using the `-g` option.
## Step 1: Compile with Debug Symbols
```
gcc -g program.c -o program
```
Here:
* `gcc` → compiler
* `-g` → adds debugging information (source code, variables, line numbers)
* `-o program` → output executable name

Without `-g`, GDB can debug only at a limited level and source lines/variables may not be visible.
## Step 2: Invoke GDB
Basic syntax:
```
gdb executable_file
```
Example:
```
gdb ./program
```
This starts GDB and loads the executable.

Typical output:
```
GNU gdb (GDB) 14.x
Reading symbols from ./program...
(gdb)
```
`(gdb)` is the debugger prompt where commands are entered.
## Different Ways to Invoke GDB
### 1. Start GDB with an Executable
```
gdb ./a.out
```
Loads the executable for debugging.
## 2. Run Program Immediately
```
gdb --args ./program input.txt hello
```
Used when the program requires command-line arguments.

Inside GDB:
```gdb
run
```
Equivalent to:
```
./program input.txt hello
```
## 3. Attach to a Running Process
If a program is already running:

First find process ID:
```
ps aux
```
or
```
pidof program
```
Attach GDB:
```
gdb -p 1234
```
Where `1234` is the process ID.

Example:
```
gdb -p 2456
```
Useful for:
* Hung programs
* Infinite loops
* Runtime inspection
## 4. Open Core Dump File
If a program crashes and produces a core dump:
```
gdb ./program core
```
Example:
```
gdb ./demo core
```
This helps analyze crashes after execution.
## 5. Start GDB Without Program
You can start GDB first:
```
gdb
```
Then load executable:
```gdb
file ./program
```
## Common GDB Startup Options
| Command                     | Purpose                |
| --------------------------- | ---------------------- |
| `gdb ./program`             | Start debugger         |
| `gdb --args ./program arg1` | Debug with arguments   |
| `gdb -p PID`                | Attach running process |
| `gdb ./program core`        | Debug core dump        |
| `gdb`                       | Start empty debugger   |
## Example Session
Program:
```c
#include <stdio.h>
int main() {
    int x = 5;
    int y = 10;
    int z = x + y;
    printf("%d\n", z);
    return 0;
}
```
Compile:
```
gcc -g add.c -o add
```
Invoke debugger:
```
gdb ./add
```
Set breakpoint:
```gdb
break main
run
```
Step execution:
```gdb
next
print x
print y
```
Output:
```
$1 = 5
$2 = 10
```

# Getting Started with gdb
**GDB (GNU Debugger)** is a tool used to debug programs by controlling execution, checking variables, and finding errors at the source-code level.

Getting started with GDB involves:
1. Compiling the program with debug information
2. Starting GDB
3. Running the program
4. Setting breakpoints
5. Executing line by line
6. Inspecting variables
## Step 1: Write a Simple Program
Example:
```c
#include <stdio.h>
int main() {
    int a = 10;
    int b = 20;
    int sum = a + b;
    printf("Sum = %d\n", sum);
    return 0;
}
```
## Step 2: Compile with Debugging Information
Compile using `-g`:
```
gcc -g add.c -o add
```
Why `-g`?
* Adds debugging symbols
* Allows source-level debugging
* Enables variable and line inspection in GDB
## Step 3: Start GDB
Run:
```
gdb ./add
```
This launches GDB.

Output:
```
GNU gdb (GDB)
Reading symbols from ./add...
(gdb)
```
`(gdb)` is the debugger prompt.
## Step 4: Set a Breakpoint
A **breakpoint** pauses execution.

Example:
```gdb
break main
```
Output:
```
Breakpoint 1 at 0x1149: file add.c, line 4.
```
This tells GDB to stop at `main()`.
## Step 5: Run the Program
Execute:
```gdb
run
```
Output:
```
Starting program: ./add
Breakpoint 1, main () at add.c:4
4       int a = 10;
```
Execution pauses at the breakpoint.
## Step 6: Execute Step by Step
### `next` (`n`)
Executes the next line.
```gdb
next
```
Example:
```
5       int b = 20;
```
Continue:
```gdb
next
next
```
## `step` (`s`)
Enters function calls.

Example:
```gdb
step
```
If a function exists:
```c
add();
```
`step` goes inside the function.
## Step 7: Print Variables
Check variable values:
```gdb
print a
print b
print sum
```
Output:
```
$1 = 10
$2 = 20
$3 = 30
```
Shortcut:
```gdb
p sum
```
## Step 8: Continue Execution
Resume program:
```gdb
continue
```
or
```gdb
c
```
Output:
```
Sum = 30
Program exited normally.
```
## Step 9: List Source Code
View source lines:
```gdb
list
```
or
```gdb
l
```
Example output:
```
1   #include <stdio.h>
2
3   int main() {
4       int a = 10;
5       int b = 20;
6       int sum = a + b;
```
## Step 10: Quit GDB
Exit debugger:
```gdb
quit
```
Shortcut:
```gdb
q
```
## Basic GDB Commands
| Command      | Short Form | Purpose            |
| ------------ | ---------: | ------------------ |
| `break main` |   `b main` | Set breakpoint     |
| `run`        |        `r` | Start program      |
| `next`       |        `n` | Execute next line  |
| `step`       |        `s` | Step into function |
| `continue`   |        `c` | Resume execution   |
| `print var`  |    `p var` | Display variable   |
| `list`       |        `l` | Show source code   |
| `quit`       |        `q` | Exit GDB           |
## Complete Example Session
```
gcc -g add.c -o add
gdb ./add
```
Inside GDB:
```gdb
break main
run
next
next
print a
print b
print sum
continue
quit
```
## Execution Flow
```
Compile → Start GDB → Breakpoint → Run →
Step Through Code → Inspect Variables →
Continue → Quit
```

# Examining and Changing Memory
In GDB, **examining memory** means viewing the contents stored at a memory location, and **changing memory** means modifying data during program execution for debugging and testing.

This is useful to:
* Inspect variables stored in memory
* Debug pointer-related problems
* Verify memory contents
* Modify values during execution without recompiling
## 1. Examining Memory
GDB uses the command:
```gdb
x
```
Syntax:
```gdb
x/FMT ADDRESS
```
Where:
* `x` → examine memory
* `FMT` → display format
* `ADDRESS` → memory location

General form:
```gdb
x /nfu address
```
Where:
* `n` → number of units to display
* `f` → format (decimal, hex, char, etc.)
* `u` → unit size
## Memory Display Formats
| Format | Meaning          |
| ------ | ---------------- |
| `x`    | Hexadecimal      |
| `d`    | Decimal          |
| `u`    | Unsigned decimal |
| `c`    | Character        |
| `s`    | String           |
| `i`    | Instruction      |
| `t`    | Binary           |
## Unit Sizes
| Symbol | Size                 |
| ------ | -------------------- |
| `b`    | byte (1 byte)        |
| `h`    | halfword (2 bytes)   |
| `w`    | word (4 bytes)       |
| `g`    | giant word (8 bytes) |
## Example Program
```c
#include <stdio.h>
int main() {
    int x = 100;
    int y = 200;
    printf("%d %d\n", x, y);
    return 0;
}
```
Compile:
```
gcc -g test.c -o test
```
Start GDB:
```
gdb ./test
```
Set breakpoint and run:
```gdb
break main
run
```
## Example 1: Examine Variable Address
Print address:
```gdb
print &x
```
Output:
```
$1 = (int *) 0x7fffffffe3bc
```
Examine memory:
```gdb
x/d &x
```
Output:
```
0x7fffffffe3bc: 100
```
Display in hexadecimal:
```gdb
x/x &x
```
Output:
```
0x7fffffffe3bc: 0x64
```
## Example 2: Examine Multiple Memory Locations
Display 4 words:
```gdb
x/4dw &x
```
Meaning:
* `4` → four values
* `d` → decimal
* `w` → word size

Example output:
```
0x7fffffffe3bc: 100 200 0 0
```
## Example 3: Examine String
Program:
```c
char str[] = "Hello";
```
View string:
```gdb
x/s str
```
Output:
```
"Hello"
```
## Example 4: Examine Instructions
Display machine instructions:
```gdb
x/10i main
```
Shows 10 instructions near `main()`.
## 2. Changing Memory
GDB allows changing values using:
```gdb
set
```
Syntax:
```gdb
set variable variable_name = value
```
or
```gdb
set {type}address = value
```
## Example 1: Change Variable Value
Suppose:
```c
int x = 100;
```
View:
```gdb
print x
```
Output:
```
$1 = 100
```
Modify:
```gdb
set variable x = 500
```
Check again:
```gdb
print x
```
Output:
```
$2 = 500
```
The program now uses `500` instead of `100`.
## Example 2: Change Memory Directly
Using address:
```gdb
print &x
```
Suppose:
```
0x7fffffffe3bc
```
Modify memory:
```gdb
set {int}0x7fffffffe3bc = 300
```
Verify:
```gdb
x/d 0x7fffffffe3bc
```
Output:
```
300
```
## Example 3: Modify Character Array
Program:
```c
char name[] = "Linux";
```
Change character:
```gdb
set name[0] = 'M'
```
Result:
```
Minux
```
## Useful Commands Summary
| Command                 | Purpose             |
| ----------------------- | ------------------- |
| `x/x address`           | Examine in hex      |
| `x/d address`           | Examine decimal     |
| `x/s address`           | Examine string      |
| `x/10i main`            | Show instructions   |
| `print &var`            | Address of variable |
| `set variable x = 10`   | Change variable     |
| `set {int}addr = value` | Modify memory       |
## Example Debugging Session
```gdb
break main
run
print x
x/d &x
set variable x = 500
print x
continue
```
Program output changes accordingly.

# Debug info Libraries
**Debug info libraries** are special versions of libraries that contain **debugging symbols (debug information)**. These symbols help a debugger like GDB show:
* Source code line numbers
* Function names
* Variable names and values
* Stack traces inside library code

Without debug information, GDB can still run, but library internals appear incomplete or difficult to understand.
## Why Are Debug Info Libraries Needed?
Suppose your program crashes inside a library function such as:
```c
printf()
malloc()
strcpy()
```
These functions belong to the C standard library, often implemented by glibc.

If debug symbols are missing:
```
#0 0x00007ffff7a52340 in ?? ()
```
You may only see memory addresses or `??`.

With debug info libraries installed:
```
#0 strcpy() at strcpy.c:54
#1 main() at test.c:15
```
Now GDB shows:
* Function names
* Source file names
* Exact line numbers

This makes debugging much easier.
## What Is Included in Debug Information?
Debug libraries usually contain:
* Symbol tables
* Function names
* Variable information
* Source file references
* Line-number mappings

These are generated during compilation using:
```
gcc -g program.c -o program
```
The `-g` option embeds debugging information.
## Example Without Debug Info
Program:
```c
#include <stdio.h>
int main() {
    char *ptr = NULL;
    printf("%s", ptr);
    return 0;
}
```
Compile:
```
gcc -g demo.c -o demo
```
Run GDB:
```
gdb ./demo
run
backtrace
```
Output may look like:
```
#0 0x7f2345abc in ?? ()
#1 0x7f2345def in ?? ()
#2 main() at demo.c:5
```
The library internals are unclear.
## Example With Debug Info Libraries
After installing debug information for system libraries:
```
#0 vfprintf() at vfprintf.c:120
#1 printf() at printf.c:32
#2 main() at demo.c:5
```
Now you can debug inside library code.
## Installing Debug Info Libraries (Linux)
Different Linux distributions provide separate debug packages.
### On Debian/Ubuntu
For GNU C Library debug symbols:
```
sudo apt install libc6-dbg
```
This installs debugging information for glibc.
## On Fedora / RHEL
```
sudo dnf debuginfo-install glibc
```
or
```
sudo yum debuginfo-install glibc
```
## Viewing Shared Libraries in GDB
Inside GDB:
```gdb
info sharedlibrary
```
This shows loaded libraries.

Example:
```
0x00007ffff7dd6000  libc.so.6
0x00007ffff7fc1000  ld-linux.so
```
## When Debug Info Libraries Are Useful
You need them when:
* Program crashes inside library functions
* Debugging segmentation faults
* Understanding system calls and runtime behavior
* Tracing memory errors in `malloc()` or `free()`
* Investigating deep stack traces
## Difference: Normal Library vs Debug Library
| Feature             | Normal Library | Debug Info Library |
| ------------------- | -------------: | -----------------: |
| Runs program        |              ✔ |                  ✔ |
| Function names      |        Limited |               Full |
| Source line info    |              ✘ |                  ✔ |
| Variable inspection |        Limited |                  ✔ |
| Better backtrace    |              ✘ |                  ✔ |

# Using gdb with a Running Process
Sometimes a program is already running, and instead of starting it inside GDB, you want to **attach GDB to the running process** to inspect what it is doing.

This is useful when:
* A program hangs or freezes
* CPU usage becomes very high
* The program enters an infinite loop
* You want to inspect memory or variables of a running process
* A server or long-running application must be debugged without restarting

With GDB, this is done using the **process ID (PID)**.
## Step 1: Find the Process ID (PID)
Before attaching GDB, find the process ID.

Using `ps`:
```
ps aux
```
Example output:
```
user   2456  2.0  ...
```
Here:
```
2456
```
is the PID.

You can also use:
```
pidof myprogram
```
Example:
```
2456
```
Or use:
```
top
```
to monitor running processes.
## Step 2: Attach GDB to the Running Process
Syntax:
```
gdb -p PID
```
Example:
```
gdb -p 2456
```
Output:
```
Attaching to process 2456
Reading symbols from ./myprogram...
(gdb)
```
GDB pauses the process and gains control.
## Example Program
Program running in infinite loop:
```c 
#include <stdio.h>
#include <unistd.h>
int main() {
    int count = 0;
    while (1) {
        printf("%d\n", count++);
        sleep(1);
    }
    return 0;
}
```
Compile:
```
gcc -g loop.c -o loop
```
Run program:
```
./loop
```
Find PID:
```
pidof loop
```
Suppose:
```
2456
```
Attach:
```
gdb -p 2456
```
## Step 3: Inspect the Running Program
### See Current Execution Point
```gdb 
where
```
or
```gdb
backtrace
```
Example:
```
#0 sleep()
#1 main() at loop.c:10
```
This shows the program is currently sleeping.
## View Variables
```gdb
print count
```
Output:
```
$1 = 20
```
## Examine Memory
```gdb 
x/d &count
```
Output:
```
20
```
## Change Variable Value
Modify runtime behavior:
```gdb 
set variable count = 100
```
Check:
```gdb
print count
```
Output:
```
$2 = 100
```
Program continues using new value.
## Step 4: Continue Execution
Resume process:
```gdb 
continue
```
or
```gdb
c
```
The process continues normally.
## Step 5: Detach from Process
To stop debugging **without terminating the program**:
```gdb 
detach
```
Output:
```
Detaching from program...
```
The process keeps running.

Exit GDB:
```gdb 
quit
```
## Step 6: Kill the Process (Optional)
Terminate inside GDB:
```gdb 
kill
```
This stops the running process.
## Important Commands
| Command             | Purpose                   |
| ------------------- | ------------------------- |
| `gdb -p PID`        | Attach to running process |
| `backtrace` / `bt`  | Show execution stack      |
| `print variable`    | View variable             |
| `set variable x=10` | Modify variable           |
| `continue`          | Resume process            |
| `detach`            | Leave process running     |
| `quit`              | Exit GDB                  |
| `kill`              | Terminate process         |
## Typical Workflow
```
Run Program
      ↓
Find PID
      ↓
gdb -p PID
      ↓
Inspect Variables / Stack / Memory
      ↓
Modify if Needed
      ↓
continue or detach
```
## Advantages of Debugging a Running Process
* No need to restart the application
* Useful for servers and daemons
* Helps analyze hangs and infinite loops
* Runtime memory inspection
* Allows live debugging

# Using gdb to Autopsy a Crash
**Autopsy a crash** means analyzing a program **after it has crashed** to determine:
* Why the crash happened
* Which line caused it
* What function was executing
* What variable values existed at the time of failure

This is done using a **core dump** (also called a core file).

A **core dump** is a snapshot of a program’s memory and execution state at the moment of the crash. It can be examined later using GDB.
## What Is a Core Dump?
When a program crashes (for example, due to a segmentation fault), Linux can save:
* Program memory
* Registers
* Stack information
* Process state

into a file called a **core dump**.

Instead of debugging while the program is running, you analyze the crash afterward.
## Step 1: Enable Core Dumps
By default, Linux may disable core dump generation.

Check limit:
```
ulimit -c
```
If output is:
```
0
```
core dumps are disabled.

Enable them:
```
ulimit -c unlimited
```
Verify:
```
ulimit -c
```
Output:
```
unlimited
```
Now Linux can generate core files after a crash.
## Step 2: Create a Program That Crashes
Example:
```c
#include <stdio.h>
int main() {
    int *ptr = NULL;
    *ptr = 10;     // crash
    return 0;
}
```
Compile with debugging symbols:
```
gcc -g crash.c -o crash
```
The `-g` option is important because it preserves debugging information.
## Step 3: Run the Program
Execute:
```
./crash
```
Example output:
```
Segmentation fault (core dumped)
```
A core file may be created:
```
core
```
or
```
core.2456
```
depending on system settings.

Check:

# Debugging Libraries - ElectricFence
Electric Fence (often written as **ElectricFence** or **libefence**) is a **debugging library** used to detect **memory-related bugs** in C/C++ programs.

It helps find problems such as:
* Buffer overflows (writing beyond allocated memory)
* Buffer underflows (writing before allocated memory)
* Accessing freed memory
* Illegal memory access

Electric Fence is especially useful for debugging programs that use:
```c
malloc()
calloc()
realloc()
free()
```
## Why Electric Fence Is Needed
Memory bugs are often hard to detect.

Example:
```c
#include <stdlib.h>
int main() {
    int *ptr = malloc(5 * sizeof(int));
    ptr[10] = 100;      // illegal access
    free(ptr);
    return 0;
}
```
Here:
```
ptr[10]
```
writes outside allocated memory.

Normally, the program may:
* Crash later
* Behave unpredictably
* Show no immediate error

With Electric Fence, the program crashes **exactly where the illegal access happens**, making debugging easier.
## How Electric Fence Works
Electric Fence places a **protected memory page** near allocated memory.

Conceptually:
```
Allocated Memory | Protected Page
```
If the program accesses memory beyond limits:
```
Segmentation Fault
```
is triggered immediately.

This helps locate the bug precisely.
## Installing Electric Fence
On Debian/Ubuntu:
```
sudo apt install electric-fence
```
On Fedora/RHEL:
```
sudo dnf install ElectricFence
```
The library installed is generally:
```
libefence
```
## Compiling with Electric Fence
Link the program using:
```
gcc program.c -lefence -g -o program
```
Where:
* `-lefence` → links Electric Fence library
* `-g` → adds debugging symbols

Example:
```
gcc overflow.c -lefence -g -o overflow
```
## Example Program
Program with memory overflow:
```c
#include <stdio.h>
#include <stdlib.h>
int main() {
    int *ptr;
    ptr = malloc(5 * sizeof(int));
    ptr[7] = 10;     // overflow
    free(ptr);
    return 0;
}
```
Compile:
```
gcc -g overflow.c -lefence -o overflow
```
Run:
```
./overflow
```
Output may be:
```
Segmentation fault
```
Instead of failing silently, Electric Fence forces an immediate crash at the faulty memory access.
## Using with GDB
Run inside GDB:
```
gdb ./overflow
```
Execute:
```gdb
run
```
Example:
```
Program received signal SIGSEGV,
Segmentation fault.
main () at overflow.c:10
10      ptr[7] = 10;
```
Now the exact line causing overflow is identified.
## Detecting Use-After-Free Errors
Example:
```c
#include <stdlib.h>
int main() {
    int *ptr = malloc(sizeof(int));
    free(ptr);
    *ptr = 10;       // invalid access
    return 0;
}
```
Without Electric Fence:
```
May or may not crash
```
With Electric Fence:
```
Immediate segmentation fault
```
This helps detect **dangling pointer access**.
## Advantages of Electric Fence
* Detects memory corruption early
* Immediate crash at offending line
* Easy integration with GDB
* Useful for pointer debugging
* Helps locate heap overflows and invalid memory access
## Limitations
* Can increase memory usage
* Slows execution
* Mainly detects heap memory errors (`malloc/free`)
* Less advanced than modern tools

Today, many developers also use:
* Valgrind for runtime memory analysis
* AddressSanitizer for compiler-assisted memory debugging
## Difference: Normal Execution vs Electric Fence
| Normal Program                    | With Electric Fence          |
| --------------------------------- | ---------------------------- |
| Memory corruption may stay hidden | Immediate detection          |
| Crash may happen later            | Crash at exact faulty access |
| Hard to debug                     | Easier debugging             |

# Debugging with Valgrind
Valgrind is a powerful debugging and profiling tool used to detect **memory-related bugs** and runtime issues in C/C++ programs.

It is commonly used to find:
* Memory leaks
* Invalid memory access
* Buffer overflows/underflows
* Use-after-free errors
* Uninitialized variable usage
* Double `free()` errors

Unlike Electric Fence, Valgrind does **not immediately crash the program**. Instead, it monitors memory usage and reports problems.
## Why Use Valgrind?
Memory bugs are difficult to detect because programs may:
* Crash much later
* Produce incorrect results
* Work sometimes and fail sometimes

Example:
```c
#include <stdlib.h>
int main() {
    int *ptr = malloc(sizeof(int));
    free(ptr);
    *ptr = 10;       // illegal access
    return 0;
}
```
Without debugging tools:
```
Program may or may not crash
```
With Valgrind:
```
Invalid write detected
```
## Installing Valgrind
On Debian/Ubuntu:
```
sudo apt install valgrind
```
On Fedora/RHEL:
```
sudo dnf install valgrind
```
## Compile Program with Debug Symbols
Compile using `-g`:
```
gcc -g test.c -o test
```
This gives meaningful source line information.
## Running Valgrind
Basic syntax:
```
valgrind ./program
```
Example:
```
valgrind ./test
```
Most memory debugging uses the default tool:
```
Memcheck
```
Equivalent command:
```
valgrind --tool=memcheck ./test
```
## Example 1: Detecting Invalid Memory Access
Program:
```c
#include <stdlib.h>
int main() {
    int *ptr = malloc(sizeof(int));
    free(ptr);
    *ptr = 10;
    return 0;
}
```
Compile:
```
gcc -g invalid.c -o invalid
```
Run:
```
valgrind ./invalid
```
Output (simplified):
```
Invalid write of size 4
   at main (invalid.c:8)
Address 0x... is inside a block of size 4 free'd
```
Meaning:
* Invalid write occurred
* File and line number shown
* Memory had already been freed
## Example 2: Detecting Memory Leaks
Program:
```c
#include <stdlib.h>
int main() {
    int *ptr;
    ptr = malloc(100);
    return 0;
}
```
Problem:
```
Memory allocated but never freed
```
Run:
```
valgrind ./test
```
Output:
```
HEAP SUMMARY:
in use at exit: 100 bytes

LEAK SUMMARY:
definitely lost: 100 bytes
```
This means memory leaked.

Fix:
```c
free(ptr);
```
## Example 3: Detecting Uninitialized Variables
Program:
```c
#include <stdio.h>
int main() {
    int x;
    printf("%d\n", x);
    return 0;
}
```
Run:
```
valgrind ./test
```
Output:
```
Conditional jump or move depends on uninitialised value(s)
```
Meaning:
```
x used before initialization
```
## Example 4: Detecting Double Free
Program:
```c
#include <stdlib.h>
int main() {
    int *ptr = malloc(sizeof(int));
    free(ptr);
    free(ptr);
    return 0;
}
```
Output:
```
Invalid free()
```
Valgrind identifies double deallocation.
## Useful Options
### Full Leak Report
```
valgrind --leak-check=full ./program
```
Shows detailed leak information.
## Track Origins of Uninitialized Values
```
valgrind --track-origins=yes ./program
```
Helps find where uninitialized values came from.
## Verbose Memory Information
```
valgrind -v ./program
```
## Typical Workflow
```
Write Program
      ↓
Compile with -g
      ↓
Run with Valgrind
      ↓
Inspect Errors
      ↓
Fix Code
      ↓
Run Again
```
## Example Session
Compile:
```
gcc -g leak.c -o leak
```
Run:
```
valgrind --leak-check=full ./leak
```
Example output:
```
HEAP SUMMARY:
in use at exit: 100 bytes

LEAK SUMMARY:
definitely lost: 100 bytes
```
Fix code:
```c
free(ptr);
```
Run again:
```
All heap blocks were freed -- no leaks are possible
```
## Common Error Types Reported by Valgrind
| Error               | Meaning                    |
| ------------------- | -------------------------- |
| Invalid read        | Reading bad memory         |
| Invalid write       | Writing outside memory     |
| Memory leak         | Allocated memory not freed |
| Use after free      | Access after `free()`      |
| Double free         | Memory freed twice         |
| Uninitialized value | Variable used before init  |
## Advantages of Valgrind
* Excellent memory debugging
* Detects leaks automatically
* No source-code modification needed
* Gives line numbers and file names
* Works well with GDB
## Limitations
* Slower execution (sometimes much slower)
* Higher memory usage
* Mainly for debugging, not production
## Electric Fence vs Valgrind
| Feature          | Electric Fence |  Valgrind |
| ---------------- | -------------: | --------: |
| Detects overflow |              ✔ |         ✔ |
| Memory leaks     |              ✘ |         ✔ |
| Use-after-free   |              ✔ |         ✔ |
| Immediate crash  |              ✔ |         ✘ |
| Detailed reports |        Limited | Excellent |

# Profiling for Performance
**Profiling for performance** means analyzing a program to determine:
* Which functions consume the most CPU time
* Where execution time is spent
* Which parts of code are slow or inefficient
* How to optimize performance

Instead of guessing why a program is slow, profiling provides **measured performance data**.

In Linux system programming, profiling is commonly done using tools such as:
* gprof
* perf
* Valgrind (with profiling tools like Callgrind)
## Why Profiling Is Needed
Suppose a program feels slow.

Without profiling:
```
Guessing which function is slow
```
With profiling:
```
Function A → 80% CPU time
Function B → 5% CPU time
Function C → 2% CPU time
```
Now optimization becomes focused and efficient.

Typical reasons:
* Slow execution
* High CPU usage
* Performance bottlenecks
* Large loops or expensive functions
* System-level optimization
## 1. Profiling Using Gprof
gprof measures function execution time and call relationships.
### Step 1: Compile with Profiling Option
Use `-pg`:
```
gcc -pg test.c -o test
```
`-pg` enables profiling support.
## Step 2: Run Program
Execute normally:
```
./test
```
This generates:
```
gmon.out
```
which stores profiling information.
## Step 3: Analyze Profile
Run:
```
gprof ./test gmon.out
```
Example output:
```
Flat profile:

time   seconds   calls  function
80.0     0.80      10   calculate()
15.0     0.15       5   process()
5.0      0.05       1   main()
```
Meaning:
* `calculate()` consumes most CPU time
* Optimization should focus there
## Example Program
```c
#include <stdio.h>
void slow() {
    long i;
    for(i = 0; i < 1000000000; i++);
}
int main() {
    slow();
    return 0;
}
```
Compile:
```
gcc -pg slow.c -o slow
```
Run:
```
./slow
```
Analyze:
```
gprof ./slow gmon.out
```
Output shows:
```
slow() consumed most time
```
## 2. Profiling Using Perf
perf is a powerful Linux profiler.

It measures:
* CPU usage
* Cache misses
* Function hotspots
* System calls
* Hardware performance counters
### Record Performance
Run:
```
perf record ./program
```
This records execution data.
## View Report
```
perf report
```
Example output:
```
80% slow_function
10% malloc
5% printf
```
This identifies bottlenecks.
## 3. Profiling with Valgrind (Callgrind)
Valgrind includes **Callgrind**, which profiles function execution.

Run:
```
valgrind --tool=callgrind ./program
```
Output file generated:
```
callgrind.out.*
```
Analyze using:
```
callgrind_annotate callgrind.out.*
```
Shows:
* Function calls
* CPU cost
* Call hierarchy
## Common Profiling Metrics
| Metric            | Meaning                    |
| ----------------- | -------------------------- |
| CPU time          | Time spent executing       |
| Function calls    | Number of calls            |
| Call graph        | Caller–callee relationship |
| Cache misses      | Memory inefficiency        |
| Execution hotspot | Slow code area             |
## Profiling Workflow
```
Write Program
      ↓
Compile with profiling support
      ↓
Run profiler
      ↓
Collect performance data
      ↓
Identify bottlenecks
      ↓
Optimize slow code
```
## Example Optimization
Before profiling:
```c
for(i=0; i<1000000000; i++)
```
takes long time.

Profiler says:
```
90% time spent here
```
Optimization:
```c
Use better algorithm
Reduce loop count
Avoid repeated computation
```
After optimization:
```
Execution faster
```
## Important Commands
| Command                     | Purpose            |
| --------------------------- | ------------------ |
| `gcc -pg file.c`            | Enable profiling   |
| `gprof program gmon.out`    | Analyze profile    |
| `perf record ./program`     | Record performance |
| `perf report`               | Show report        |
| `valgrind --tool=callgrind` | Profile execution  |
## Advantages of Profiling
* Finds real bottlenecks
* Improves performance scientifically
* Saves optimization effort
* Helps CPU and memory optimization
* Useful for large applications
