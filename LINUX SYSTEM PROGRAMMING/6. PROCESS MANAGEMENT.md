# What a Process is 
A **process** is a program that is currently being executed by the operating system.

In simple terms:
> **Program = passive (stored on disk)**

> **Process = active (running in memory)**

For example:
* A C program stored as `hello.c` or compiled executable `a.out` is just a **program**.
* When you run it (`./a.out`), the operating system loads it into memory and executes it — now it becomes a **process**.
## Characteristics of a Process
A process contains:
1. **Program Code (Text Segment)**

   The actual instructions to execute.
2. **Data Section**

   Global and static variables.
3. **Heap**

   Dynamically allocated memory (`malloc()` in C).
4. **Stack**

   Function calls, local variables, return addresses.
5. **Program Counter**

   Keeps track of the next instruction to execute.
6. **CPU Registers**

   Store temporary execution data.
### Process Memory Layout
```
+------------------+
|     Stack        | ← Local variables, function calls
+------------------+
|      Heap        | ← Dynamic memory (malloc)
+------------------+
|   Data Segment   | ← Global/static variables
+------------------+
|   Text Segment   | ← Program instructions
+------------------+
```
### Example
Consider this C program:
```c
#include <stdio.h>
int main() {
    printf("Hello World\n");
    return 0;
}
```
* Before execution → it is a **program**.
* After running → OS creates a **process** for it.
### Process States
A process generally moves through these states:
```
New → Ready → Running → Waiting/Blocked → Ready → Terminated
```
* **New** → Process is being created
* **Ready** → Waiting for CPU
* **Running** → Executing on CPU
* **Waiting/Blocked** → Waiting for I/O/event
* **Terminated** → Finished execution
### In Linux
Every process has a unique identifier called **PID (Process ID)**.

You can see running processes using:
```
ps
```
or
```
top
```
Example:
```
$ ps
PID   TTY     TIME   CMD
1234  pts/0   00:00  bash
5678  pts/0   00:00  a.out
```
Here `5678` is the PID of the running process.

# Process Relationships
In an operating system, processes can have relationships with other processes, mainly through **creation and hierarchy**.

The most important relationship is:
> **Parent Process ↔ Child Process**

When one process creates another process, the creator is called the **parent process**, and the created process is called the **child process**.

In Linux, this is commonly done using the `fork()` system call.
## Parent and Child Process
```
Parent Process
      |
   fork()
      |
      ├── Child Process 1
      ├── Child Process 2
      └── Child Process 3
```
Example:
* A shell (`bash`) runs a command.
* The shell creates a new process to execute the command.

Example hierarchy:
```
init/systemd (PID 1)
        |
      bash
        |
      gcc
```
Here:
* `systemd` is parent of `bash`
* `bash` is parent of `gcc`

You can view process hierarchy in Linux using:
```
pstree
```
Example output:
```
systemd─┬─bash───vim
        └─sshd───bash
```
## Types of Process Relationships
### 1. Parent Process
A process that creates another process.

Example:
* Shell creates a program.
```c
pid_t pid = fork();
```
The process calling `fork()` becomes the parent.
### 2. Child Process
The newly created process.

After `fork()`:
* Parent and child execute independently.
* Child gets a copy of the parent’s memory space.
## Sibling Processes
Processes created by the same parent are called **siblings**.
```
        Parent
       /   |   \
    Child1 Child2 Child3
```
Here, Child1, Child2, and Child3 are sibling processes.
## Orphan Process
If a parent terminates before the child finishes, the child becomes an **orphan process**.

Then the OS assigns it to the process with PID 1 (usually `systemd` or `init`).
```
Parent (dies)
      X
      |
    Child  → adopted by PID 1
```
Example:
* Parent exits.
* Child still running.

The child continues execution.
## Zombie Process
A zombie process is a process that has completed execution but still has an entry in the process table because the parent has not collected its exit status.
```
Parent running
      |
   Child exits
      |
 Zombie (waiting for parent to read status)
```
This happens when the parent does not call functions like:
```c
wait();
```
or
```c
waitpid();
```
Zombie consumes almost no memory but occupies a process table entry.
## Process Tree in Linux
Linux processes are arranged in a hierarchical tree.

At the top:

systemd (or historically `init`) → PID 1

Example:
```
systemd (PID 1)
 ├── sshd
 │    └── bash
 │         └── vim
 └── chrome
```
## Important System Calls in Process Relationships
| System Call     | Purpose                |
| --------------- | ---------------------- |
| `fork()`        | Create child process   |
| `exec()` family | Replace process image  |
| `wait()`        | Parent waits for child |
| `getpid()`      | Get process ID         |
| `getppid()`     | Get parent process ID  |
## Example Program
```c
#include <stdio.h>
#include <unistd.h>
int main() {
    pid_t pid = fork();
    if(pid == 0) {
        printf("Child Process\n");
    } else {
        printf("Parent Process\n");
    }
    return 0;
}
```
Output:
```
Parent Process
Child Process
```
After `fork()`:
* Parent and child run concurrently.
* Both continue from the next instruction after `fork()`.

# Create a Child Process
In Linux, a child process is created using the `fork()` system call.
## What is `fork()`?
`fork()` creates a **new process** (child process) by duplicating the calling process (parent process).

Syntax:
```c
pid_t fork(void);
```
After `fork()`:
* A **child process** is created.
* Parent and child execute **independently**.
* Both continue execution from the statement immediately after `fork()`.
### Return Value of `fork()`
| Return Value | Meaning                                         |
| ------------ | ----------------------------------------------- |
| `0`          | Executing in child process                      |
| `> 0`        | Executing in parent process (returns child PID) |
| `< 0`        | Error in process creation                       |
## Process Creation Flow
```
Parent Process
      |
    fork()
      |
 ┌────┴────┐
 |         |
Parent    Child
```
## Example Program: Create a Child Process
```c
#include <stdio.h>
#include <unistd.h>
int main() {
    pid_t pid;
    pid = fork();
    if(pid < 0) {
        printf("Fork failed\n");
    }
    else if(pid == 0) {
        printf("Child Process\n");
        printf("PID = %d\n", getpid());
        printf("Parent PID = %d\n", getppid());
    }
    else {
        printf("Parent Process\n");
        printf("PID = %d\n", getpid());
        printf("Child PID = %d\n", pid);
    }
    return 0;
}
```
### Sample Output
```
Parent Process
PID = 1200
Child PID = 1201

Child Process
PID = 1201
Parent PID = 1200
```
### Explanation
1. Parent process starts execution.
2. `fork()` is called.
3. OS creates a child process.
4. Execution splits into two flows:
* **Parent process** (`pid > 0`)
* **Child process** (`pid == 0`)

Both execute concurrently.
### How `fork()` Works
```
Before fork()

Parent Process
     |
   fork()

After fork()

Parent Process -------->
Child Process  -------->
```
The child gets a copy of:
* Program code
* Variables
* File descriptors
* Environment

But parent and child have **different PIDs**.
## Example with Variable Copy
```c
#include <stdio.h>
#include <unistd.h>
int main() {
    int x = 10;
    pid_t pid = fork();
    if(pid == 0) {
        x = 20;
        printf("Child x = %d\n", x);
    } else {
        printf("Parent x = %d\n", x);
    }
    return 0;
}
```
Possible output:
```
Parent x = 10
Child x = 20
```
Explanation:
* Child receives a copy of `x`.
* Changing `x` in child does not affect parent.
### Compile and Run
```
gcc child.c -o child
./child
```
### Related System Calls
| System Call | Purpose                 |
| ----------- | ----------------------- |
| `fork()`    | Create child process    |
| `getpid()`  | Get current process ID  |
| `getppid()` | Get parent process ID   |
| `wait()`    | Parent waits for child  |
| `exec()`    | Execute another program |

# Doing Something Else
After a child process is created using `fork()`, the **parent process and child process execute independently**.

This means:
> The parent can continue doing some other work while the child performs a different task.

The operating system scheduler decides when each process runs.
## Example Concept
```
Parent Process
      |
    fork()
      |
 ┌────┴────┐
 |         |
Parent    Child
(doing    (doing
something  another
 else)      task)
```
For example:
* Parent process → prints numbers
* Child process → prints alphabets

Both run concurrently.
## Example Program
```c 
#include <stdio.h>
#include <unistd.h>
int main() {
    pid_t pid = fork();
    if(pid == 0) {
        // Child process
        for(int i = 1; i <= 5; i++) {
            printf("Child Process: %d\n", i);
            sleep(1);
        }
    }
    else {
        // Parent process doing something else
        for(char ch = 'A'; ch <= 'E'; ch++) {
            printf("Parent Process: %c\n", ch);
            sleep(1);
        }
    }
    return 0;
}
```
### Possible Output
```
Parent Process: A
Child Process: 1
Parent Process: B
Child Process: 2
Parent Process: C
Child Process: 3
Parent Process: D
Child Process: 4
Parent Process: E
Child Process: 5
```
Output order may change because the OS schedules processes independently.
## Explanation
1. Parent starts execution.
2. `fork()` creates a child process.
3. Both processes execute simultaneously:
   * Child executes its own block (`pid == 0`)
   * Parent executes another block (`pid > 0`)
4. Parent does **something else** instead of waiting for the child.

This demonstrates **concurrent execution**.
### Execution Flow
```
Parent Process
      |
    fork()
      |
 ┌────┴────┐
 |         |
Child      Parent
Task       Different Task
```
## Key Point
After `fork()`:
* Parent and child run independently.
* Parent does not automatically wait for child.
* To make the parent wait, use:
```c 
wait(NULL);
```
Without `wait()`, parent can continue “doing something else”.

# Related `execve()` Functions
In Linux, the **`exec` family of functions** is used to execute a new program inside an existing process.

These functions are related to `execve()` and are wrappers around it.
> `fork()` creates a new process
> `exec()` replaces the current process image with a new program
## What `execve()` Does
When a process calls `execve()`:
* The current program is replaced by another program.
* Process ID (PID) remains the same.
* Code, data, stack, and heap are replaced.

Flow:
```
Process Running
      |
   execve()
      |
New Program Executes
(Same PID)
```
## Syntax of `execve()`
```c 
#include <unistd.h>
int execve(const char *pathname,
           char *const argv[],
           char *const envp[]);
```
Parameters:
* `pathname` → program path
* `argv[]` → command-line arguments
* `envp[]` → environment variables
## `exec` Family Functions
Common related functions:
| Function   | Description             |
| ---------- | ----------------------- |
| `execl()`  | Argument list           |
| `execv()`  | Argument vector (array) |
| `execlp()` | Search command in PATH  |
| `execvp()` | Vector + PATH search    |
| `execle()` | Custom environment      |
| `execve()` | Core system call        |
## 1. `execl()` – List Version
Uses arguments as a list.

Syntax:
```c 
execl(path, arg0, arg1, ..., NULL);
```
Example:
```c 
#include <stdio.h>
#include <unistd.h>
int main() {
    printf("Before execl\n");
    execl("/bin/ls", "ls", "-l", NULL);
    printf("After execl\n");
    return 0;
}
```
Output:
```
Before execl
(list of files)
```
`After execl` is not printed because the process image is replaced.
## 2. `execv()` – Vector Version
Arguments passed as an array.

Syntax:
```c 
execv(path, argv);
```
Example:
```c 
#include <stdio.h>
#include <unistd.h>
int main() {
    char *args[] = {"ls", "-l", NULL};
    execv("/bin/ls", args);
    return 0;
}
```
## 3. `execlp()` – PATH Search + List
Searches executable in PATH.

Syntax:
```c 
execlp(file, arg0, arg1, ..., NULL);
```
Example:
```c 
execlp("ls", "ls", "-l", NULL);
```
No need for `/bin/ls`.
## 4. `execvp()` – PATH Search + Vector
Most commonly used with `fork()`.

Syntax:
```c 
execvp(file, argv);
```
Example:
```c 
#include <stdio.h>
#include <unistd.h>
int main() {
    char *args[] = {"ls", "-l", NULL};
    execvp("ls", args);
    return 0;
}
```
## 5. `execle()` – Custom Environment
Allows passing environment variables.

Syntax:
```c 
execle(path, arg0, ..., NULL, envp);
```
Example:
```c 
char *env[] = {"USER=test", NULL};
execle("/bin/printenv",
       "printenv",
       NULL,
       env);
```
## 6. `execve()` – Base Function
All `exec` functions internally use `execve()`.

Example:
```c 
#include <unistd.h>
int main() {
    char *args[] = {"ls", "-l", NULL};
    execve("/bin/ls", args, NULL);
    return 0;
}
```
## Relationship Between `fork()` and `exec()`
Usually used together:
```
Parent
   |
 fork()
   |
 Child
   |
 exec()
   |
 New Program
```
Example:
```c 
#include <stdio.h>
#include <unistd.h>
int main() {
    pid_t pid = fork();
    if(pid == 0) {
        execlp("ls", "ls", "-l", NULL);
    }
    else {
        printf("Parent continues\n");
    }
    return 0;
}
```
Explanation:
* Parent creates child.
* Child executes `ls`.
* Parent continues independently.
## Difference Between `fork()` and `exec()`
| `fork()`                 | `exec()`               |
| ------------------------ | ---------------------- |
| Creates new process      | Replaces process image |
| New PID created          | Same PID retained      |
| Parent and child execute | Old program removed    |
## Important Rule
If `exec()` succeeds:
> It never returns.

Only returns on failure (`-1`).

Example:
```c 
printf("Before exec\n");
execlp("ls", "ls", NULL);
printf("After exec\n");
```
`After exec` never prints.

# Wait For a Child
In Linux, a parent process can wait for its child process to finish execution using the `wait()` or `waitpid()` system calls.

Why?

After `fork()`:
* Parent and child run independently.
* Sometimes the parent must wait until the child finishes before continuing.

This is called **process synchronization**.
## `wait()` System Call
Syntax:
```c
#include <sys/types.h>
#include <sys/wait.h>
pid_t wait(int *status);
```
Purpose:
> Suspends the parent process until one of its child processes terminates.

When the child finishes:
* Parent resumes execution.
* Child exit status can be collected.
* Prevents zombie processes.
## Process Flow
```
Parent Process
      |
    fork()
      |
 ┌────┴────┐
 |         |
Child     Parent
Running   wait()
   |         |
 exits <-----|
      Parent resumes
```
## Example Program: Parent Waits for Child
```c
#include <stdio.h>
#include <unistd.h>
#include <sys/wait.h>
int main() {
    pid_t pid = fork();
    if(pid == 0) {
        // Child process
        printf("Child Process Running\n");
        sleep(3);
        printf("Child Process Finished\n");
    }
    else {
        // Parent waits
        wait(NULL);
        printf("Parent Process Resumed\n");
    }
    return 0;
}
```
### Sample Output
```
Child Process Running
Child Process Finished
Parent Process Resumed
```
### Explanation
1. Parent creates child using `fork()`.
2. Child executes and sleeps for 3 seconds.
3. Parent calls:
```c
wait(NULL);
```
4. Parent blocks (stops temporarily).
5. Child finishes.
6. Parent resumes execution.
## `wait(NULL)`
```c
wait(NULL);
```
`NULL` means:
> Parent is waiting, but not interested in the child’s exit status.
## Getting Child Exit Status
Example:
```c
#include <stdio.h>
#include <unistd.h>
#include <sys/wait.h>
int main() {
    int status;
    pid_t pid = fork();
    if(pid == 0) {
        printf("Child Process\n");
        return 5;
    }
    else {
        wait(&status);
        printf("Parent Process\n");
        if(WIFEXITED(status)) {
            printf("Exit Status = %d\n",
                   WEXITSTATUS(status));
        }
    }
    return 0;
}
```
Possible output:
```
Child Process
Parent Process
Exit Status = 5
```
### Explanation
Macros used:
* `WIFEXITED(status)` → checks normal termination
* `WEXITSTATUS(status)` → gets exit value
## `waitpid()` System Call
`waitpid()` gives more control.

Syntax:
```c
pid_t waitpid(pid_t pid,
              int *status,
              int options);
```
Purpose:
* Wait for a specific child process
* Non-blocking wait possible

Example:
```c
waitpid(pid, NULL, 0);
```
Waits for the child with given PID.
## Why `wait()` Is Important
Without `wait()`:
* Parent may terminate first.
* Child may become an orphan.
* Child termination may create a zombie process until status is collected.

With `wait()`:
* Proper synchronization
* Cleanup of child process entry
## Execution Comparison
Without `wait()`:
```
Parent starts
Child starts
Parent exits
Child exits
```
With `wait()`:
```
Parent starts
Child starts
Child exits
Parent resumes
```
## Related Functions
| Function    | Purpose                 |
| ----------- | ----------------------- |
| `fork()`    | Create child process    |
| `wait()`    | Wait for child          |
| `waitpid()` | Wait for specific child |
| `exit()`    | Terminate process       |

# More Precise Waiting
The `wait()` system call waits for **any child process** to terminate.

Sometimes we need more control, such as:
* Wait for a **specific child process**
* Wait **without blocking**
* Wait for child state changes

For this, Linux provides:
```c
waitpid()
```
## Why `waitpid()`?
`waitpid()` gives **more precise control** over child process waiting than `wait()`.

Example situation:
```
Parent
 ├── Child 1
 ├── Child 2
 └── Child 3
```
With `wait()`:
> Parent waits for **any** child.

With `waitpid()`:
> Parent can wait for a **particular child**.
## Syntax
```c 
#include <sys/types.h>
#include <sys/wait.h>
pid_t waitpid(pid_t pid,
              int *status,
              int options);
```
### Parameters
| Parameter | Meaning                 |
| --------- | ----------------------- |
| `pid`     | Which child to wait for |
| `status`  | Exit status of child    |
| `options` | Behavior control        |
## `pid` Values
| Value  | Meaning                              |
| ------ | ------------------------------------ |
| `> 0`  | Wait for specific child PID          |
| `-1`   | Wait for any child (like `wait()`)   |
| `0`    | Wait for child in same process group |
| `< -1` | Wait for a process group             |
Example:
```c 
waitpid(pid, NULL, 0);
```
Waits for the child with that PID.
## Example: Waiting for a Specific Child
```c 
#include <stdio.h>
#include <unistd.h>
#include <sys/wait.h>
int main() {
    pid_t pid = fork();
    if(pid == 0) {
        printf("Child Running\n");
        sleep(3);
        printf("Child Finished\n");
    }
    else {
        printf("Parent Waiting...\n");
        waitpid(pid, NULL, 0);
        printf("Parent Resumed\n");
    }
    return 0;
}
```
### Sample Output
```
Parent Waiting...
Child Running
Child Finished
Parent Resumed
```
Explanation:
* Parent waits for the child whose PID is stored in `pid`.
* Parent resumes only after child finishes.
## Non-Blocking Wait (`WNOHANG`)
Normally:
```c 
waitpid(pid, NULL, 0);
```
blocks the parent.

Using:
```c 
waitpid(pid, NULL, WNOHANG);
```
makes it **non-blocking**.

Meaning:
> Parent checks child status and continues immediately if child has not terminated.
### Example
```c 
#include <stdio.h>
#include <unistd.h>
#include <sys/wait.h>
int main() {
    pid_t pid = fork();
    if(pid == 0) {
        sleep(5);
        printf("Child Finished\n");
    }
    else {
        printf("Parent Checking...\n");
        if(waitpid(pid, NULL, WNOHANG) == 0) {
            printf("Child still running\n");
        }
        printf("Parent continues work\n");
    }
    return 0;
}
```
Possible output:
```
Parent Checking...
Child still running
Parent continues work
Child Finished
```
Explanation:
* Parent does not block.
* Child continues independently.
## Important `options`
| Option       | Meaning                 |
| ------------ | ----------------------- |
| `0`          | Normal blocking wait    |
| `WNOHANG`    | Non-blocking            |
| `WUNTRACED`  | Return if child stopped |
| `WCONTINUED` | Return if child resumed |
## Difference Between `wait()` and `waitpid()`
| `wait()`            | `waitpid()`             |
| ------------------- | ----------------------- |
| Waits for any child | Wait for specific child |
| Less control        | More precise control    |
| Always blocking     | Can be non-blocking     |
Equivalent:
```c 
wait(NULL);
```
same as:
```c 
waitpid(-1, NULL, 0);
```
## Process Flow
```
Parent
   |
fork()
   |
Child running
   |
waitpid(pid)
   |
Child exits
   |
Parent resumes
```

# Changing Priority/Nice
In Linux, process scheduling priority can be influenced using the **nice value**.

A **nice value** determines how much CPU time a process gets relative to other processes.
> A process with a lower nice value gets higher CPU priority.

> A process with a higher nice value gets lower CPU priority.
## Priority vs Nice
* **Priority** → Internal scheduler priority used by the OS
* **Nice value** → User-controlled value that affects scheduling priority

Think of it like:
> “How nice is this process to others?”

A process with a high nice value is “nicer” because it gives CPU time to other processes.
## Nice Value Range
```
-20  --------------------->  +19
Highest Priority              Lowest Priority
```
| Nice Value | Meaning          |
| ---------- | ---------------- |
| `-20`      | Highest priority |
| `0`        | Default priority |
| `+19`      | Lowest priority  |
Important:
* Lower nice → more CPU preference
* Higher nice → less CPU preference
## Viewing Process Priority
Use:
```
ps -l
```
Example:
```
F S UID PID PPID C PRI NI ADDR SZ WCHAN TTY TIME CMD
0 S 1000 1234 1200 0 80  0 -    -  pts/0 00:00 bash
```
Where:
* `PRI` → scheduler priority
* `NI` → nice value

You can also use:
```
top
```
## `nice()` System Call
Used to change a process’s nice value.

Syntax:
```c
#include <unistd.h>
int nice(int increment);
```
Parameter:
* `increment` → value added to current nice value

Example:
```c
nice(5);
```
If current nice is `0`:
```
0 + 5 = 5
```
The process becomes lower priority.
## Example Program
```c
#include <stdio.h>
#include <unistd.h>
int main() {
    int ret;
    ret = nice(5);
    printf("New nice value = %d\n", ret);
    while(1);
    return 0;
}
```
Explanation:
* Process priority is reduced.
* Infinite loop keeps process running so it can be observed.

Compile and run:
```
gcc nice.c -o nice
./nice
```
Check priority:
```
ps -l
```
## `getpriority()` and `setpriority()`
Linux provides more precise control.

Header:
```c
#include <sys/resource.h>
```
### `getpriority()`
Gets current priority.

Syntax:
```c
int getpriority(int which, id_t who);
```
Example:
```c
int p = getpriority(PRIO_PROCESS, 0);
printf("%d\n", p);
```
`0` means current process.
## `setpriority()`
Changes priority.

Syntax:
```c
int setpriority(int which,
                id_t who,
                int prio);
```
Example:
```c
setpriority(PRIO_PROCESS, 0, 10);
```
Sets current process nice value to `10`.
## Example Program Using `setpriority()`
```c
#include <stdio.h>
#include <sys/resource.h>
int main() {
    int ret;
    ret = setpriority(PRIO_PROCESS, 0, 10);
    if(ret == 0)
        printf("Priority changed\n");
    printf("Nice value = %d\n",
           getpriority(PRIO_PROCESS, 0));
    return 0;
}
```
## Important Permission Rule
Increasing priority (lowering nice value) usually requires root privileges.

Example:
```c
setpriority(PRIO_PROCESS, 0, -10);
```
May fail for normal users.

But lowering priority:
```c
setpriority(PRIO_PROCESS, 0, 10);
```
is usually allowed.
## Command-Line Utilities
Start a process with a nice value:
```
nice -n 10 ./program
```
Change priority of running process:
```
renice 5 -p 1234
```
Where:
* `5` → new nice value
* `1234` → process PID
## Example Scenario
```
Video Rendering Process → nice = 10
System Service         → nice = -5
```
System service gets CPU preference.
## Summary
```
Lower nice value  → Higher priority
Higher nice value → Lower priority
```
| Function        | Purpose              |
| --------------- | -------------------- |
| `nice()`        | Increment nice value |
| `getpriority()` | Read priority        |
| `setpriority()` | Change priority      |

# Real Time Priority
In Linux, **real-time priority** is used for processes that require **immediate and predictable CPU response**.

Examples:
* Audio/video streaming
* Industrial control systems
* Robotics
* Embedded systems
* Real-time data acquisition

Unlike normal processes (which use **nice values**), real-time processes use **real-time scheduling priorities**.
> Real-time processes get CPU preference over normal processes.
## Normal Priority vs Real-Time Priority
| Normal Process                  | Real-Time Process                |
| ------------------------------- | -------------------------------- |
| Uses nice value (`-20` to `19`) | Uses real-time priority (`1–99`) |
| Fair CPU sharing                | Immediate CPU scheduling         |
| Time-sharing scheduler          | Real-time scheduler              |
Rule:
```
Real-Time Process Priority > Normal Process Priority
```
Meaning:
> A real-time process runs before ordinary processes.
## Real-Time Priority Range
```
1 ----------------------> 99
Low RT Priority           High RT Priority
```
* `1` → Lowest real-time priority
* `99` → Highest real-time priority

Important:
> Even a real-time process with priority `1` usually outranks normal processes.
## Real-Time Scheduling Policies
Linux supports mainly two real-time scheduling policies:
### 1. `SCHED_FIFO` (First In First Out)
* Highest-priority runnable process executes first.
* Runs until:
  * It blocks,
  * Terminates,
  * Or a higher-priority process preempts it.

Flow:
```
Priority 90 → Running
Priority 80 → Waiting
Priority 70 → Waiting
```
If priority `95` arrives:
```
Priority 95 → Running
Priority 90 → Preempted
```
Characteristics:
* No time slicing
* Strict priority scheduling
## 2. `SCHED_RR` (Round Robin)
Similar to `SCHED_FIFO`, but:
> Processes with the same priority share CPU in time slices.

Example:
```
Priority 50
 ├── Process A
 ├── Process B
 └── Process C
```
Execution:
```
A → B → C → A → B → C
```
Characteristics:
* Time quantum used
* Fair sharing among same-priority tasks
## 3. `SCHED_OTHER`
Default Linux scheduler.
* Used for ordinary processes
* Influenced by nice value
## Setting Real-Time Priority
Linux provides:
```c 
sched_setscheduler()
```
Syntax:
```c 
#include <sched.h>
int sched_setscheduler(pid_t pid,
                       int policy,
                       const struct sched_param *param);
```
Parameters:
* `pid` → process ID (`0` for current process)
* `policy` → scheduling policy
* `param` → priority
## Example Program
```c 
#include <stdio.h>
#include <sched.h>
int main() {
    struct sched_param param;
    param.sched_priority = 50;
    if(sched_setscheduler(0,
                           SCHED_FIFO,
                           &param) == -1) {
        perror("sched_setscheduler");
        return 1;
    }
    printf("Real-time priority set\n");
    while(1);
    return 0;
}
```
Explanation:
* Sets current process priority to `50`
* Uses `SCHED_FIFO`
* Infinite loop keeps process alive
## Compile and Run
```
gcc rt.c -o rt
sudo ./rt
```
Why `sudo`?
> Real-time scheduling usually requires root privileges.
## Getting Scheduling Information
Use:
```c
sched_getscheduler(0);
```
Get priority:
```c 
sched_getparam(0, &param);
```
## Linux Command-Line Tools
View process scheduling:
```
ps -eo pid,comm,pri,ni,cls
```
Example:
```
PID CMD      PRI NI CLS
100 audio    99  - FF
200 bash     20   0 TS
```
Where:
* `FF` → FIFO real-time
* `RR` → Round Robin
* `TS` → Time sharing
## Important Difference: Nice vs Real-Time
| Nice Priority           | Real-Time Priority     |
| ----------------------- | ---------------------- |
| `-20` to `19`           | `1–99`                 |
| Normal scheduling       | Real-time scheduling   |
| Relative CPU preference | Immediate CPU access   |
| `nice()`                | `sched_setscheduler()` |
## Example Scenario
```
Audio Playback Process → RT priority 80
Video Encoder         → nice 5
Text Editor           → nice 0
```
Audio playback gets CPU first to avoid glitches.
## Key Point
> Real-time processes can starve normal processes if misused.

A badly written real-time task may make the system unresponsive.
