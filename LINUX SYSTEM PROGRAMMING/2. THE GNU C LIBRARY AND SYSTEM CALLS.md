# Library Goals
A **library** in an operating system/programming environment is a collection of pre-written functions, routines, or reusable code that helps programmers perform tasks without writing everything from scratch.
## Goals of a Library
1. **Code Reusability**
   * Avoid rewriting common functions repeatedly.
   * Example: mathematical operations, string handling, file handling.
2. **Reduce Development Time**
   * Programmers can use ready-made functions instead of implementing them again.
   * Faster software development.
3. **Modularity**
   * Divide programs into reusable modules.
   * Makes programs easier to manage and maintain.
4. **Ease of Programming**
   * Provides simple interfaces for complex operations.
   * Example: file I/O functions (`fopen()`, `fread()`, `fwrite()`) from the C standard library.
5. **Improve Reliability**
   * Library functions are tested and optimized.
   * Reduces bugs in application development.
6. **Code Maintenance**
   * Changes or bug fixes can be done in one place (library) instead of modifying multiple programs.
7. **Portability**
   * Same library functions work across different systems with minimal changes.
   * Example: standard C library functions work on Linux, Windows, etc.
8. **Efficient Resource Usage**
   * Shared libraries reduce memory usage because multiple programs can use the same library code.
### Types of Libraries
* **Static Library** – Linked during compilation (e.g., `.a` files in Linux).
* **Dynamic/Shared Library** – Loaded during execution (e.g., `.so` files in Linux).
### Example
In C programming:
```c
#include <stdio.h>
int main() {
    printf("Hello World\n");
    return 0;
}
```
Here, `printf()` comes from the C standard library (`stdio.h`).

In Linux system programming, libraries help access system resources like files, processes, memory, networking, and threads easily.

# Library Standards
**Library standards** are a set of rules and specifications followed while designing and using libraries so that programs become portable, reliable, and compatible across systems.
## Goals of Library Standards
* Ensure **uniformity** in function behavior.
* Improve **portability** of programs across systems.
* Provide **consistent interfaces** for programmers.
* Increase **reliability and compatibility**.
### Common Library Standards in C/Linux
1. ANSI C Standard
   * Defines standard C library functions.
   * Includes functions for input/output, memory management, string handling, etc.
   * Example headers: `stdio.h`, `stdlib.h`, `string.h`.
2. ISO C Standard
   * International standard for the C language.
   * Ensures portability across different operating systems.
3. POSIX Standard
   * Defines standards for operating-system interfaces in Unix/Linux systems.
   * Provides APIs for processes, threads, file handling, signals, IPC, etc.
   * Example headers: `unistd.h`, `pthread.h`, `fcntl.h`.
4. GNU C Library
   * Standard C library implementation used in Linux systems.
   * Implements both C standard and POSIX functions.
### Important Characteristics of Library Standards
* **Consistency** – Same function behaves the same everywhere.
* **Portability** – Code runs on different platforms with minimal changes.
* **Compatibility** – Programs can work across systems.
* **Maintainability** – Easier to update and debug.
### Example
`printf()` behaves the same on different systems because it follows standardized library rules:
```c
#include <stdio.h>
int main() {
    printf("Welcome to Linux Programming\n");
    return 0;
}
```
Here, `printf()` follows C library standards, ensuring consistent output behavior across systems.

In Linux System Programming, **POSIX standards** are especially important because they define how programs interact with the Linux kernel and operating system services.

# GNU C Library - glibc
The **GNU C Library (glibc)** is the standard C library implementation used in most Linux systems. It provides the functions needed for system programming and application development in C and C++.

It acts as a **bridge between application programs and the Linux kernel** by providing standard APIs for system calls and common programming tasks.
## Features of glibc
1. **Implements Standard C Library**
   * Provides functions defined by ANSI/ISO C standards.
   * Examples:
     * `printf()`
     * `scanf()`
     * `malloc()`
     * `free()`
     * `strcpy()`
2. **Supports POSIX Standards**
   * Provides Linux/Unix system programming APIs.
   * Example functions:
     * `fork()` → Create process
     * `exec()` → Execute program
     * `open()`, `read()`, `write()` → File operations
     * `pthread_create()` → Thread creation
3. **System Call Interface**
   * Applications usually do not communicate directly with the kernel.
   * glibc provides wrapper functions for system calls.

Example flow:
```
Application Program
        ↓
      glibc
        ↓
 Linux Kernel (System Calls)
        ↓
     Hardware
```
4. **Memory Management Support**
   * Dynamic memory allocation:
   ```c
   malloc(), calloc(), realloc(), free()
   ```
5. **String and Utility Functions**
   * String handling:
   ```c
   strlen(), strcpy(), strcmp()
   ```
6. **Input/Output Support**
   * Standard I/O:
   ```c
   printf(), scanf(), fopen(), fread(), fwrite()
   ```
### Example Program Using glibc
```c 
#include <stdio.h>
#include <stdlib.h>
int main() {
    int *ptr = malloc(sizeof(int));
    *ptr = 100;
    printf("Value = %d\n", *ptr);
    free(ptr);
    return 0;
}
```
Here:
* `malloc()` and `free()` come from glibc.
* `printf()` is also provided by glibc.
### Advantages of glibc
* Standardized and portable.
* Provides ready-made functions.
* Simplifies Linux system programming.
* Supports C and POSIX APIs.
* Efficient and optimized.
### Why glibc is Important in Linux Programming?
Without glibc, programmers would need to directly invoke kernel system calls for many operations. glibc makes programming easier by offering simple, reusable functions that internally interact with the Linux kernel.

# Library Functions vs. System Calls
| Feature     | Library Functions                                                      | System Calls                                                        |
| ----------- | ---------------------------------------------------------------------- | ------------------------------------------------------------------- |
| Definition  | Predefined functions provided by libraries for programming convenience | Functions used to request services from the operating system kernel |
| Purpose     | Simplify programming by providing reusable functions                   | Allow user programs to interact with the kernel                     |
| Execution   | Executes in **user space**                                             | Executes in **kernel space**                                        |
| Speed       | Faster (usually less overhead)                                         | Slower (requires mode switching between user and kernel mode)       |
| Dependency  | May or may not use system calls internally                             | Directly interact with OS kernel                                    |
| Provided By | Libraries such as GNU C Library                                        | Operating System Kernel                                             |
| Portability | More portable                                                          | OS dependent                                                        |
| Examples    | `printf()`, `scanf()`, `strlen()`, `malloc()`                          | `open()`, `read()`, `write()`, `fork()`, `exec()`                   |
## What are Library Functions?
Library functions are **ready-made functions** provided by libraries to make programming easier.

Example:
```c
#include <stdio.h>
int main() {
    printf("Hello Linux\n");
    return 0;
}
```
Here, `printf()` is a library function provided by glibc.
### What are System Calls?
System calls are the interface between a user program and the operating system kernel. They are used to access hardware and OS services.

Example:
```c
#include <unistd.h>
int main() {
    write(1, "Hello Linux\n", 12);
    return 0;
}
```
Here, `write()` sends data to the kernel to display output.
### Working Flow
```
User Program
     ↓
Library Function (glibc)
     ↓
System Call
     ↓
Linux Kernel
     ↓
Hardware
```
Example:
* `printf()` → library function
* Internally, `printf()` may use `write()` system call to display output.
### Key Difference
* **Library Function** → Convenience function for programmers.
* **System Call** → Request to kernel for OS-level service.

**Simple Analogy:**

Library function = Receptionist who helps you fill a request.

System call = Actual request reaching the manager (kernel) for action.

# Using System Calls
A **system call** is used by a program to request services from the operating system kernel, such as file handling, process creation, memory management, and device access.

In Linux, system calls are generally accessed through functions provided by the GNU C Library.
## Steps to Use System Calls
1. Include the required header file.
2. Call the system call function.
3. Check the return value for success or failure.
4. Handle errors if needed.
### General Syntax
```c
return_type system_call(arguments);
```
Example:
```c
read(fd, buffer, size);
```
Where:
* `fd` → File descriptor
* `buffer` → Memory to store data
* `size` → Number of bytes to read
### Commonly Used System Calls
| System Call | Purpose                    |
| ----------- | -------------------------- |
| `open()`    | Open a file                |
| `read()`    | Read data from file/device |
| `write()`   | Write data to file/device  |
| `close()`   | Close a file               |
| `fork()`    | Create a new process       |
| `exec()`    | Execute a new program      |
| `wait()`    | Wait for child process     |
| `getpid()`  | Get process ID             |
### Example 1: Using `write()` System Call
```c
#include <unistd.h>
int main() {
    write(1, "Hello Linux\n", 12);
    return 0;
}
```
Explanation:
* `1` → Standard output (screen)
* `"Hello Linux\n"` → Message to print
* `12` → Number of bytes to write

Flow:
```
Program
   ↓
write() System Call
   ↓
Kernel
   ↓
Terminal Output
```
### Example 2: Using `open()`, `write()`, and `close()`
```c
#include <fcntl.h>
#include <unistd.h>
int main() {
    int fd;
    fd = open("sample.txt", O_WRONLY | O_CREAT, 0644);
    write(fd, "Linux Programming", 17);
    close(fd);
    return 0;
}
```
Explanation:
* `open()` → Opens/creates file
* `write()` → Writes data into file
* `close()` → Closes the file
### Return Values
Most system calls return:
* **Success** → Non-negative value
* **Failure** → `-1`

Example:
```c
int fd = open("abc.txt", O_RDONLY);
if (fd == -1) {
    // error handling
}
```
### Why Use System Calls?
* Access OS services directly
* File handling
* Process management
* Communication with devices
* Memory and resource management

In Linux system programming, system calls are the **main interface between user applications and the kernel**.

# Handling Errors with errno
In Linux system programming, when a **system call** or some library functions fail, they usually return an error value and set a special variable called **`errno`**.

`errno` helps identify **why the error occurred**.
## What is `errno`?
`errno` is a global integer variable defined in:
```c 
#include <errno.h>
```
* It stores the **error number** when a function fails.
* Most system calls return `-1` on failure and set `errno`.
### Working of `errno`
```
System Call Fails
        ↓
Return Value = -1
        ↓
errno is set
        ↓
Check error reason
```
### Example: Using `errno`
```c 
#include <stdio.h>
#include <fcntl.h>
#include <errno.h>
int main() {
    int fd;
    fd = open("test.txt", O_RDONLY);
    if (fd == -1) {
        printf("Error Number: %d\n", errno);
    }
    return 0;
}
```
Explanation:
* If `test.txt` does not exist:
  * `open()` returns `-1`
  * `errno` gets an error code.
### Common Error Handling Functions
1. **`perror()`**
   * Prints a user-defined message along with the error description.

Example:
```c 
#include <stdio.h>
#include <fcntl.h>
int main() {
    int fd = open("test.txt", O_RDONLY);
    if (fd == -1) {
        perror("Open Failed");
    }
    return 0;
}
```
Output example:
```
Open Failed: No such file or directory
```
2. **`strerror()`**
   * Converts `errno` value into a readable string.

Example:
```c 
#include <stdio.h>
#include <fcntl.h>
#include <errno.h>
#include <string.h>
int main() {
    int fd = open("test.txt", O_RDONLY);
    if (fd == -1) {
        printf("Error: %s\n", strerror(errno));
    }
    return 0;
}
```
### Common `errno` Values
| Error Code | Meaning                   |
| ---------- | ------------------------- |
| `ENOENT`   | No such file or directory |
| `EACCES`   | Permission denied         |
| `EBADF`    | Bad file descriptor       |
| `ENOMEM`   | Out of memory             |
| `EEXIST`   | File already exists       |
| `EINVAL`   | Invalid argument          |
### Example Flow
```
open("file.txt")
      ↓
Failure (-1)
      ↓
errno = ENOENT
      ↓
perror()/strerror()
      ↓
Display error message
```
### Important Notes
* Always check the return value before using `errno`.
* `errno` is meaningful **only after a function fails**.
* Include `errno.h` to use `errno`.
### Why `errno` is Important?
* Helps debug programs.
* Gives the exact reason for failure.
* Makes programs more reliable through proper error handling.

# Making Sense of errno
`errno` is an integer variable that stores an **error number** when a system call or library function fails. However, numeric error values alone are hard to understand.

**Making sense of `errno`** means converting the error number into a **meaningful, human-readable message**.

For example:
```c 
errno = 2
```
The value `2` alone is unclear. But it actually means:
```
No such file or directory
```
## Ways to Understand (`errno`) Errors
### 1. Using `perror()`
`perror()` prints a custom message followed by the error description associated with `errno`.

Example:
```c
#include <stdio.h>
#include <fcntl.h>
int main() {
    int fd;
    fd = open("abc.txt", O_RDONLY);
    if (fd == -1) {
        perror("Open failed");
    }
    return 0;
}
```
Output:
```
Open failed: No such file or directory
```
Explanation:
* `Open failed` → User-defined message
* `No such file or directory` → Meaning of `errno`
### 2. Using `strerror()`
`strerror()` converts the `errno` value into a string.

Example:
```c 
#include <stdio.h>
#include <fcntl.h>
#include <errno.h>
#include <string.h>
int main() {
    int fd;
    fd = open("abc.txt", O_RDONLY);
    if (fd == -1) {
        printf("Error No: %d\n", errno);
        printf("Meaning: %s\n", strerror(errno));
    }
    return 0;
}
```
Example Output:
```
Error No: 2
Meaning: No such file or directory
```
### Common `errno` Meanings
| Error Name | Meaning                   |
| ---------- | ------------------------- |
| `ENOENT`   | No such file or directory |
| `EACCES`   | Permission denied         |
| `ENOMEM`   | Not enough memory         |
| `EBADF`    | Bad file descriptor       |
| `EINVAL`   | Invalid argument          |
| `EEXIST`   | File already exists       |
### Program Flow
```
System Call
     ↓
Failure
     ↓
errno gets value
     ↓
perror() / strerror()
     ↓
Readable error message
```
### Why Make Sense of `errno`?
* Easier debugging
* Understand exact failure reason
* Improve program reliability
* Better error reporting to users

**Simple idea:**

`errno` gives a number, but `perror()` and `strerror()` explain what that number means.

# Using Strace
`strace` is a Linux debugging utility used to **trace system calls and signals** made by a program.

It helps programmers understand:
* Which **system calls** a program is making
* What files are opened
* What data is read/written
* Why a program fails
* Error codes returned (`errno`)

In Linux system programming, `strace` is very useful for **debugging system call behavior**.
## Syntax
```
strace command
```
Example:
```
strace ls
```
This traces all system calls made by the `ls` command.
### Example Output
```
openat(AT_FDCWD, "/etc/ld.so.cache", O_RDONLY) = 3
read(3, "...", 832) = 832
close(3) = 0
write(1, "file1 file2\n", 12) = 12
```
Explanation:
* `openat()` → Opens a file
* `read()` → Reads data
* `close()` → Closes file
* `write()` → Prints output
### Tracing a Program
Suppose program:
```c 
#include <unistd.h>
int main() {
    write(1, "Hello Linux\n", 12);
    return 0;
}
```
Compile:
```
gcc test.c -o test
```
Run with `strace`:
```
strace ./test
```
Output (simplified):
```
write(1, "Hello Linux\n", 12) = 12
Hello Linux
exit_group(0) = ?
```
Explanation:
* `write()` system call prints text
* `exit_group()` terminates process
### Useful `strace` Options
| Command                      | Purpose                         |
| ---------------------------- | ------------------------------- |
| `strace program`             | Trace all system calls          |
| `strace -o file.txt program` | Save output to a file           |
| `strace -e open ls`          | Trace only specific system call |
| `strace -p PID`              | Attach to running process       |
| `strace -c program`          | Summary of system calls         |
Example:
```
strace -e write ./test
```
Traces only `write()` calls.
### Debugging with `strace`
If a file cannot be opened:
```c 
open("data.txt", O_RDONLY);
```
Run:
```
strace ./program
```
Possible output:
```
openat(AT_FDCWD, "data.txt", O_RDONLY) = -1 ENOENT
```
Meaning:
* `ENOENT` → No such file or directory

So `strace` helps identify the exact reason for failure.
### Flow of `strace`
```
Program Execution
        ↓
strace monitors system calls
        ↓
Shows system calls + return values
        ↓
Helps debugging
```
### Advantages of `strace`
* Debug system call failures
* Understand program behavior
* Diagnose permission issues
* View `errno` errors
* Monitor file, process, and memory operations

`strace` is one of the most important debugging tools in Linux system programming because it shows **how a program communicates with the kernel**.
