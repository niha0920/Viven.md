# Why Use Libraries?
A **library** is a collection of pre-written functions that programmers can reuse instead of writing everything from scratch.

In Linux system programming, libraries (especially the C standard library, called glibc) make programming easier, faster, and portable.
## 1. Code Reusability
Instead of writing common functionality yourself, you reuse existing tested functions.

Example:
```c
#include <stdio.h>
int main() {
    printf("Hello World\n");
    return 0;
}
```
Here, `printf()` comes from the C library.

Without libraries, you would have to write low-level code to display text on the screen.
## 2. Saves Development Time
Libraries provide ready-made functions for:
* File handling → `fopen()`, `fread()`, `fwrite()`
* Memory management → `malloc()`, `free()`
* String handling → `strlen()`, `strcpy()`
* Math operations → `sqrt()`, `pow()`

Instead of reinventing these functions, developers use libraries and focus on application logic.
## 3. Easier Access to System Calls
In Linux, many operations ultimately use system calls, but libraries provide a simpler interface.

Example:

To write text to terminal:

Using system call:
```c
#include <unistd.h>
int main() {
    write(1, "Hello\n", 6);
    return 0;
}
```
Using library function:
```c
#include <stdio.h>
int main() {
    printf("Hello\n");
    return 0;
}
```
`printf()` internally may use the `write()` system call, but it is easier to use and provides formatting support.
## 4. Portability
Libraries hide OS-specific details.

For example:
```c
printf("Age = %d", age);
```
The same code works on different systems because the library handles low-level implementation.

This makes programs more portable.
## 5. Reliability and Optimization
Library functions are:
* Well tested
* Optimized for performance
* Less error-prone

Example:

`memcpy()` is highly optimized compared to manually copying bytes in a loop.
## 6. Modularity and Maintainability
Programs become cleaner and easier to maintain.

Instead of:
```c
// 100 lines for string comparison
```
You simply use:
```c
strcmp(str1, str2);
```
This improves readability.
## Real-Life Analogy
Think of libraries like tools in a toolbox 🧰.

If you want to tighten a screw, you use a screwdriver instead of building one yourself.

Similarly, programmers use libraries instead of writing every function from scratch.
## In Linux System Programming
Libraries (like glibc) sit between your application and the kernel:
```
Application Program
        ↓
Library Functions (glibc)
        ↓
System Calls
        ↓
Linux Kernel
```
So, libraries provide a **convenient, reusable, optimized interface to system resources**.

# Static Versus Shared 
In Linux/C programming, libraries are mainly of two types:
1. **Static Libraries**
2. **Shared (Dynamic) Libraries**

Both provide reusable code, but they differ in how the program uses them.
## 1. Static Library
A **static library** is linked to the program **at compile time**.

The required library code is copied into the executable file.
### Flow
```
Source Code
     ↓
Compiler + Static Library (.a)
     ↓
Executable (contains library code)
```
### Extension
Static libraries usually have:
```
.a
```
Example:
```
libm.a
libpthread.a
```
### Characteristics
* Library code becomes part of executable
* No dependency on external library during execution
* Executable size becomes larger
* Faster loading (library already included)
### Example
Compile using static linking:
```
gcc program.c -static -o program
```
## 2. Shared (Dynamic) Library
A **shared library** is linked **at runtime**.

The executable contains only a reference to the library, and the actual code is loaded when the program runs.
### Flow
```
Source Code
      ↓
Compiler + Shared Library (.so)
      ↓
Executable (reference only)
      ↓
Runtime Loader
      ↓
Shared Library Loaded
```
### Extension
Shared libraries usually have:
```
.so
```
Example:
```
libc.so
libm.so
```
### Characteristics
* Smaller executable size
* Library shared among many programs
* Saves memory
* Easier updates (update library once)
### Example
Normal GCC compilation generally uses shared libraries:
```
gcc program.c -o program
```
## Comparison Table
| Feature         |          Static Library |                  Shared Library |
| --------------- | ----------------------: | ------------------------------: |
| Linking Time    |            Compile time |                         Runtime |
| Extension       |                    `.a` |                           `.so` |
| Executable Size |                   Large |                           Small |
| Performance     | Slightly faster startup |          Runtime loading needed |
| Memory Usage    |                    More |                            Less |
| Dependency      |             Independent |     Depends on external library |
| Updates         |      Recompile required | Library update affects programs |
## Example
Suppose your program uses `printf()`.
### Static Linking
```
Executable
 ├── main()
 └── printf() code copied inside
```
Even if system library is removed, program works.
### Shared Linking
```
Executable
 └── Reference → libc.so
```
At runtime, OS loads the library.

If the shared library is missing, program may fail.
## Real-Life Analogy
**Static Library** → Carrying your own water bottle everywhere.

You are independent but carry extra weight.

**Shared Library** → Using a common water cooler in office.

Less weight, shared resource, but must be available.
## In Linux
Most Linux programs use **shared libraries** because they:
* Save memory
* Reduce executable size
* Allow easy updates

Static libraries are used when:
* Creating standalone executables
* Embedded systems
* Recovery tools or minimal systems

# Static Library Benefits
A **static library** provides several advantages because the library code is copied directly into the executable during compilation.
## 1. No External Dependency
The executable contains all required library code.

So, the program can run even if the library is not installed on the target system.
### Example
If your program is statically linked:
```
program
 ├── main()
 ├── printf()
 └── malloc()
```
Everything is already included.

**Benefit:** Easy deployment and portability.
## 2. Standalone Executable
The program becomes self-contained.

You can move the executable to another machine and run it without worrying about missing shared libraries.

Example:
```
./myprogram
```
No need for external `.so` files.
## 3. Faster Program Startup
Since library code is already inside the executable:
* No runtime searching for libraries
* No dynamic loading overhead

Execution starts slightly faster.
## 4. Better Reliability
The program always uses the exact version of the library it was built with.

This avoids problems caused by library updates.

Example:

A shared library update may break compatibility:
```
Program → libc.so version mismatch
```
Static linking avoids such issues.
## 5. Useful in Embedded Systems
In embedded Linux or small systems:
* Shared libraries may not exist
* Minimal runtime environment

Static linking ensures the program works independently.

This is common in:
* Firmware
* Rescue systems
* Embedded devices
## 6. Easier Distribution
You only distribute:
```
myprogram
```
Instead of:
```
myprogram
libabc.so
libxyz.so
```
Less dependency management.
## 7. More Predictable Behavior
Since the executable always uses built-in code:
* No accidental replacement of libraries
* No version conflicts

Program behavior remains consistent.
## Example Compilation
```
gcc main.c -static -o app
```
This creates a statically linked executable.
## Drawbacks (for balance)
Static libraries also have disadvantages:
* Larger executable size
* Higher memory usage
* Updating library requires recompiling
## Summary Table
| Benefit           | Explanation                  |
| ----------------- | ---------------------------- |
| Independence      | No external `.so` needed     |
| Portability       | Runs on other systems easily |
| Faster startup    | No runtime loading           |
| Reliability       | Fixed library version        |
| Embedded support  | Good for standalone systems  |
| Easy distribution | Single executable            |

# Shared Library Benefits
A **shared library** (dynamic library) is loaded at **runtime** and can be shared by multiple programs.

It offers several advantages over static libraries.
## 1. Smaller Executable Size
The executable contains only a reference to the library, not the actual code.
### Example
Instead of copying `printf()` into every executable:
```
Program → libc.so
```
Only a reference is stored.

**Benefit:** Smaller executable files.
## 2. Memory Saving
Multiple programs can share the same library in memory.

Example:
```
Program1 ─┐
Program2 ─┼── libc.so (shared in RAM)
Program3 ─┘
```
Instead of loading separate copies, Linux keeps one copy in memory.

**Benefit:** Efficient RAM usage.
## 3. Easier Updates
Update the shared library once, and all programs using it automatically benefit.

Example:
```
Old libc.so → Updated libc.so
```
Programs start using the new version without recompilation.

**Benefit:** Easier maintenance.
## 4. Reduced Disk Space Usage
Since many programs use the same shared library:
```
Program A → libc.so
Program B → libc.so
Program C → libc.so
```
Only one copy of the library is stored on disk.

**Benefit:** Saves storage space.
## 5. Faster Development and Maintenance
Developers can update or fix library code independently of applications.

For example:
* Bug fixes
* Security patches
* Performance improvements

No need to rebuild every application.
## 6. Modular Programming
Programs can be split into reusable modules.

Example:
```
Application
   ├── Graphics Library
   ├── Math Library
   └── Networking Library
```
This improves organization and reuse.
## 7. Runtime Flexibility
Libraries can be loaded dynamically when needed using functions like:
```c 
dlopen()
dlsym()
dlclose()
```
This enables:
* Plugins
* Optional features
* Runtime extension

Example:

A media player loading codec plugins dynamically.
## Example Compilation
Normally GCC links shared libraries by default:
```
gcc main.c -o app
```
To see shared dependencies:
```
ldd app
```
Example output:
```
libc.so.6 => /lib/x86_64-linux-gnu/libc.so.6
```
## Drawbacks (for balance)
Shared libraries also have disadvantages:
* Dependency on external `.so` files
* Slight runtime loading overhead
* Version mismatch issues may occur
## Summary Table
| Benefit             | Explanation             |
| ------------------- | ----------------------- |
| Small executable    | Library code not copied |
| Memory efficient    | Shared in RAM           |
| Easy updates        | Change library once     |
| Disk space saving   | One copy shared         |
| Maintainability     | Independent updates     |
| Modularity          | Reusable components     |
| Runtime flexibility | Dynamic loading support |

# Creating a Static Library
A **static library** is a collection of compiled object files packed into a single file with extension `.a`.

The general process is:
```
Source Files (.c)
        ↓
Compile to Object Files (.o)
        ↓
Create Static Library (.a)
        ↓
Link with Application
```
## Step 1: Create Source Files
Suppose we create a small math library.
### `add.c`
```c 
int add(int a, int b)
{
    return a + b;
}
```
### `sub.c`
```c 
int sub(int a, int b)
{
    return a - b;
}
```
## Step 2: Compile into Object Files
Compile source files into `.o` files.
```
gcc -c add.c
gcc -c sub.c
```
This generates:
```
add.o
sub.o
```
`-c` means **compile only** (do not link).
## Step 3: Create Static Library
Use the `ar` (archive) utility.
```
ar rcs libmath.a add.o sub.o
```
Explanation:
* `r` → Replace/add files into archive
* `c` → Create library if it doesn't exist
* `s` → Create index for faster access

Generated file:
```
libmath.a
```
Naming convention:
```
lib<name>.a
```
Example:
```
libmath.a
```
## Step 4: Create Main Program
### `main.c`
```c 
#include <stdio.h>
int add(int, int);
int sub(int, int);
int main()
{
    printf("Addition = %d\n", add(10, 5));
    printf("Subtraction = %d\n", sub(10, 5));
    return 0;
}
```
## Step 5: Link Static Library
Compile and link program:
```
gcc main.c -L. -lmath -o app
```
Explanation:
* `-L.` → Search current directory for library
* `-lmath` → Link `libmath.a`
* `-o app` → Output executable name

The linker automatically searches:
```
libmath.a
```
for:
```
-lmath
```
## Step 6: Run Program
```
./app
```
Output:
```
Addition = 15
Subtraction = 5
```
## Verify Static Linking
Use:
```
ldd app
```
If fully static, it may show:
```
not a dynamic executable
```
Or fewer shared dependencies.
## Complete Command Sequence
```
gcc -c add.c
gcc -c sub.c
ar rcs libmath.a add.o sub.o
gcc main.c -L. -lmath -o app
./app
```
## Directory Structure
```
project/
│── add.c
│── sub.c
│── add.o
│── sub.o
│── libmath.a
│── main.c
└── app
```
## Important Notes
* Static libraries use extension `.a`
* Created using `ar`
* Linked during compilation
* Code gets copied into executable

# Using Static Libraries
After creating a static library (`.a`), the next step is to **use it in a program**.

The process is:
```
Create Library (.a)
        ↓
Write Program
        ↓
Link Library During Compilation
        ↓
Run Executable
```
## Example: Using `libmath.a`
Suppose we already created:
```
libmath.a
```
containing:
* `add()`
* `sub()`
## Step 1: Create Header File (Recommended)
### `math.h`
```c
#ifndef MATH_H
#define MATH_H
int add(int a, int b);
int sub(int a, int b);
#endif
```
Why?

Instead of manually declaring function prototypes in every file, we place them in a header.
## Step 2: Write Main Program
### `main.c`
```c 
#include <stdio.h>
#include "math.h"
int main()
{
    printf("Addition = %d\n", add(10, 5));
    printf("Subtraction = %d\n", sub(10, 5));
    return 0;
}
```
## Step 3: Link the Static Library
Compile and link:
```
gcc main.c -L. -lmath -o app
```
Explanation:
| Option   | Meaning                                |
| -------- | -------------------------------------- |
| `-L.`    | Search current directory for libraries |
| `-lmath` | Link `libmath.a`                       |
| `-o app` | Create executable `app`                |
Important:
```
-lmath
```
means:
```
libmath.a
```
GCC automatically adds:
```
lib
```
and:
```
.a
```
## Step 4: Run Program
```
./app
```
Output:
```
Addition = 15
Subtraction = 5
```
## Project Structure
```
project/
│── add.c
│── sub.c
│── math.h
│── add.o
│── sub.o
│── libmath.a
│── main.c
└── app
```
## Full Build Process
### Create object files
```
gcc -c add.c
gcc -c sub.c
```
### Create static library
```
ar rcs libmath.a add.o sub.o
```
### Compile program using library
```
gcc main.c -L. -lmath -o app
```
### Run
```
./app
```
## Forcing Complete Static Linking
Normally GCC may still use shared system libraries.

To create a **fully static executable**:
```
gcc main.c -L. -lmath -static -o app
```
This links even system libraries statically.
## Verify Linking
Check dependencies:
```
ldd app
```
If fully static:
```
not a dynamic executable
```
## Common Mistakes
### 1. Forgetting `-L`
Wrong:
```
gcc main.c -lmath -o app
```
Error:
```
cannot find -lmath
```
Because linker doesn’t know library location.

Correct:
```
gcc main.c -L. -lmath -o app
```
## 2. Wrong library name
Wrong:
```
-l libmath.a
```
Correct:
```
-lmath
```
## Summary
To use a static library:
1. Create `.a` library
2. Include header file
3. Compile with `-L` and `-l`
4. Run executable

# Creating a Shared Library
A **shared library** (`.so`) is a library that is **loaded at runtime** and shared among multiple programs.

The process is:
```
Source Files (.c)
        ↓
Compile as Position Independent Code (PIC)
        ↓
Create Shared Library (.so)
        ↓
Link Program
        ↓
Load at Runtime
```
## Step 1: Create Source Files
### `add.c`
```c 
int add(int a, int b)
{
    return a + b;
}
```
### `sub.c`
```c 
int sub(int a, int b)
{
    return a - b;
}
```
## Step 2: Create Header File
### `math.h`
```c 
#ifndef MATH_H
#define MATH_H
int add(int a, int b);
int sub(int a, int b);
#endif
```
Header files provide function declarations.
## Step 3: Compile Object Files with `-fPIC`
Compile using:
```
gcc -c -fPIC add.c
gcc -c -fPIC sub.c
```
Generated:
```
add.o
sub.o
```
### Why `-fPIC`?
`-fPIC` means **Position Independent Code**.

It allows the library to be loaded at different memory locations.

Without it, shared libraries may not work correctly.
## Step 4: Create Shared Library
Use GCC:
```
gcc -shared -o libmath.so add.o sub.o
```
Generated:
```
libmath.so
```
Naming convention:
```
lib<name>.so
```
Example:
```
libmath.so
```
## Step 5: Write Main Program
### `main.c`
```c
#include <stdio.h>
#include "math.h"
int main()
{
    printf("Addition = %d\n", add(10, 5));
    printf("Subtraction = %d\n", sub(10, 5));
    return 0;
}
```
## Step 6: Compile and Link Program
Link shared library:
```
gcc main.c -L. -lmath -o app
```
Explanation:
| Option   | Meaning                  |
| -------- | ------------------------ |
| `-L.`    | Search current directory |
| `-lmath` | Link `libmath.so`        |
| `-o app` | Output executable        |
## Step 7: Set Library Path
At runtime, Linux must find `libmath.so`.

Temporarily set:
```
export LD_LIBRARY_PATH=.
```
Or:
```
export LD_LIBRARY_PATH=$LD_LIBRARY_PATH:.
```
## Step 8: Run Program
```
./app
```
Output:
```
Addition = 15
Subtraction = 5
```
## Verify Shared Dependency
Use:
```
ldd app
```
Example output:
```
libmath.so => ./libmath.so
libc.so.6 => /lib/x86_64-linux-gnu/libc.so.6
```
This confirms dynamic linking.
## Complete Command Sequence
```
gcc -c -fPIC add.c
gcc -c -fPIC sub.c
gcc -shared -o libmath.so add.o sub.o
gcc main.c -L. -lmath -o app
export LD_LIBRARY_PATH=.
./app
```
## Project Structure
```
project/
│── add.c
│── sub.c
│── math.h
│── add.o
│── sub.o
│── libmath.so
│── main.c
└── app
```
## Common Mistakes
### 1. Forgetting `-fPIC`
Wrong:
```
gcc -c add.c
```
Correct:
```
gcc -c -fPIC add.c
```
## 2. Library Not Found at Runtime
Error:
```
error while loading shared libraries:
libmath.so: cannot open shared object file
```
Fix:
```
export LD_LIBRARY_PATH=.
```
## Summary
To create a shared library:
1. Compile with `-fPIC`
2. Create `.so` using `-shared`
3. Link with `-L` and `-l`
4. Set runtime path if needed

# Using Shared Libraries
After creating a shared library (`.so`), you must **link it during compilation** and ensure Linux can **find it at runtime**.

The process is:
```
Create Shared Library (.so)
          ↓
Write Program
          ↓
Link with Library
          ↓
Set Runtime Path
          ↓
Run Program
```
## Example: Using `libmath.so`
Suppose we already created:
```
libmath.so
```
containing:
* `add()`
* `sub()`
## Step 1: Create Header File
### `math.h`
```c 
#ifndef MATH_H
#define MATH_H
int add(int a, int b);
int sub(int a, int b);
#endif
```
The header contains function declarations.
## Step 2: Write Main Program
### `main.c`
```c 
#include <stdio.h>
#include "math.h"
int main()
{
    printf("Addition = %d\n", add(10, 5));
    printf("Subtraction = %d\n", sub(10, 5));
    return 0;
}
```
## Step 3: Link Shared Library
Compile program:
```
gcc main.c -L. -lmath -o app
```
Explanation:
| Option   | Meaning                                |
| -------- | -------------------------------------- |
| `-L.`    | Search current directory for libraries |
| `-lmath` | Link `libmath.so`                      |
| `-o app` | Output executable                      |
Important:
```
-lmath
```
automatically means:
```
libmath.so
```
GCC adds:
```
lib
```
and:
```
.so
```
## Step 4: Runtime Library Search
At runtime, Linux must find:
```
libmath.so
```
If it is not in a standard library path (`/lib`, `/usr/lib`, etc.), set:
```
export LD_LIBRARY_PATH=.
```
or:
```
export LD_LIBRARY_PATH=$LD_LIBRARY_PATH:.
```
This tells Linux to search the current directory.
## Step 5: Run Program
```
./app
```
Output:
```
Addition = 15
Subtraction = 5
```
## Verify Shared Library Dependency
Use:
```
ldd app
```
Example output:
```
linux-vdso.so.1
libmath.so => ./libmath.so
libc.so.6 => /lib/x86_64-linux-gnu/libc.so.6
```
This confirms:
* `app` uses `libmath.so`
* library is dynamically loaded
## Complete Example
### Create shared library
```
gcc -c -fPIC add.c
gcc -c -fPIC sub.c
gcc -shared -o libmath.so add.o sub.o
```
### Compile application
```
gcc main.c -L. -lmath -o app
```
### Set library path
```
export LD_LIBRARY_PATH=.
```
### Run
```
./app
```
## Better Method: Embed Runtime Path (`rpath`)
Instead of setting `LD_LIBRARY_PATH` every time:
```
gcc main.c -L. -lmath -Wl,-rpath=. -o app
```
Now run directly:
```
./app
```
No need for:
```
LD_LIBRARY_PATH
```
## Common Error
Error:
```
error while loading shared libraries:
libmath.so: cannot open shared object file
```
Reason:

Linux cannot find the shared library.

Fix:
```
export LD_LIBRARY_PATH=.
```
or use:
```
-Wl,-rpath=.
```
## Static vs Shared Usage
| Feature            | Static Library | Shared Library |
| ------------------ | -------------: | -------------: |
| File               |           `.a` |          `.so` |
| Linked             |   Compile time |        Runtime |
| Runtime dependency |             No |            Yes |
| Need library path  |             No |            Yes |
## Summary
To use a shared library:
1. Create `.so` library
2. Include header file
3. Compile with `-L` and `-l`
4. Ensure runtime library path (`LD_LIBRARY_PATH` or `rpath`)
5. Run program

# Shared Library Management
**Shared library management** means handling how shared libraries (`.so`) are:
* Installed
* Located
* Loaded
* Updated
* Versioned

Linux provides mechanisms to manage shared libraries efficiently.
## How Linux Finds Shared Libraries
When a program runs, the **dynamic linker/loader** searches for required `.so` files.

Flow:
```
Executable
    ↓
Dynamic Linker (ld.so / ld-linux.so)
    ↓
Search Shared Library
    ↓
Load into Memory
```
Typical locations searched:
```
/lib
/usr/lib
/usr/local/lib
```
And paths configured in:
```
/etc/ld.so.conf
```
## Check Shared Library Dependencies
Use:
```
ldd app
```
Example output:
```
linux-vdso.so.1
libmath.so => ./libmath.so
libc.so.6 => /lib/x86_64-linux-gnu/libc.so.6
```
This shows:
* Required libraries
* Their locations
## `LD_LIBRARY_PATH`
A temporary environment variable for specifying extra library paths.

Example:
```
export LD_LIBRARY_PATH=.
```
Or:
```
export LD_LIBRARY_PATH=/home/user/lib
```
Multiple paths:
```
export LD_LIBRARY_PATH=/lib1:/lib2:/home/user/lib
```
### Check variable
```
echo $LD_LIBRARY_PATH
```
**Use case:** Testing libraries during development.
## Permanent Library Configuration
Instead of setting `LD_LIBRARY_PATH` every time:

Add path to:
```
/etc/ld.so.conf
```
Or create a file inside:
```
/etc/ld.so.conf.d/
```
Example:
```
/home/user/mylibs
```
Then run:
```
sudo ldconfig
```
This updates the shared library cache.
## `ldconfig`
`ldconfig` manages shared library cache.

Command:
```
sudo ldconfig
```
Purpose:
* Updates cache
* Creates symbolic links
* Speeds library lookup

Cache file:
```
/etc/ld.so.cache
```
Check cached libraries:
```
ldconfig -p
```
## Shared Library Versioning
Libraries often have versions.

Example:
```
libm.so
libm.so.6
libm.so.6.1
```
Typical structure:
```
libxyz.so -> libxyz.so.1
libxyz.so.1 -> libxyz.so.1.2.3
```
Benefits:
* Compatibility
* Safe upgrades
* Multiple versions can coexist
## Symbolic Links in Libraries
Example:
```
ls -l /usr/lib/libm.so*
```
Output:
```
libm.so -> libm.so.6
libm.so.6 -> libm.so.6.1
```
Why?

Programs link against:
```
libm.so
```
but actual implementation may be:
```
libm.so.6.1
```
## Runtime Path (`rpath`)
Instead of environment variables:
```
gcc main.c -L. -lmath -Wl,-rpath=. -o app
```
This embeds library path into executable.

Check:
```
readelf -d app
```
## Debugging Library Issues
### Check dependencies
```
ldd app
```
### Check cache
```
ldconfig -p
```
### Find missing symbols
```
nm libmath.so
```
### Show ELF information
```
readelf -a libmath.so
```
## Common Errors
### 1. Library not found
Error:
```
libmath.so: cannot open shared object file
```
Fix:
```
export LD_LIBRARY_PATH=.
```
or:
```
sudo ldconfig
```
## 2. Undefined symbol
Error:
```
undefined symbol: add
```
Reason:
* Missing function in library
* Wrong library version
